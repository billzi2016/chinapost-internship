# Week1 项目分析文档总览

本目录用于集中存放 `Week1：邮政客服 SFT 与 RAG 项目` 的补充分析文档，主要服务于阶段性研究整理、方法说明与报告材料撰写。相关文档既关注工程落地，也强调模型选型、训练风险和方法论说明的完整性。

## 文档构成

- `模型选型报告.md`
  - 说明本项目为什么优先选择 `Qwen2.5` 和 `gpt-oss:20b`
  - 同时给出对 `Qwen3`、`Llama 3.3 70B`、`Llama 4`、`Mistral`、`Mixtral`、`Gemma`、`DeepSeek-R1-Distill`、`InternLM2.5`、`MoE` 路线、过小模型等候选的排除理由

- `SFT训练与风险控制.md`
  - 说明在 `k` 级别中文客服数据上如何设计 SFT 训练
  - 重点回答“如何避免模型训崩 / crash / collapse，以及该看哪些指标”

## 当前阶段结论摘要

当前阶段的选型结论可以概括为三句话：

1. 主力 SFT 候选优先放在 `Qwen2.5-3B` 和 `Qwen2.5-7B`
2. `gpt-oss:20b` 可以作为更大模型上限和本地强基线，但不建议作为第一优先 SFT 主力
3. `70B` 级别模型、`Llama 4` 多模态 MoE、以及明显偏 `thinking / reasoning` 的路线，都不适合本项目当前的数据规模、中文重点和当前阶段资源约束

## 当前阶段优先选择 Qwen2.5 的主要原因

对于本项目而言，模型选型的核心标准并不是“理论性能上限最高”，而是“能否在当前数据规模、中文业务场景、SFT 实施难度与本地资源约束下，稳定完成训练、验证与展示闭环”。  
因此，模型选型优先级是：

- 中文能力
- 指令跟随能力
- 结构化输出能力，尤其是 `JSON`
- 小数据下做领域适配的稳定性
- 推理速度和部署可行性
- 是否容易完成从训练到验证到展示的闭环

从这几个维度看，`Qwen2.5` 比很多最新但更重、更复杂、或更偏 reasoning 的模型更符合当前阶段目标。
