# KeyBERT

## 1. 这个方法在项目里做什么

`KeyBERT` 是本项目里最偏“语义路线”的关键词提取方法。  
和前面的 `TF-IDF`、`TextRank` 不同，它不再主要依赖词频或共现网络，而是依赖语义嵌入。

对应代码位置：

- `/Users/bizi/Desktop/邮政实习/week1/stats/keybert_analysis.py`

## 2. 原理

`KeyBERT` 的核心思想是：

1. 先把整段文本编码成一个语义向量
2. 再把候选关键词或短语也编码成语义向量
3. 计算候选项和整段文本之间的语义相似度
4. 相似度越高，越可能是能代表主题的关键词

这和 `TF-IDF` 最大的不同在于：

- `TF-IDF` 看的是“出现多少次”
- `KeyBERT` 看的是“语义上像不像这篇文本在说的东西”

因此，从理论上说，`KeyBERT` 更不容易被：

- 频率很高但语义空泛的词

干扰。

## 3. 本项目里具体怎么做

### 3.1 输入仍然来自 documents

和前面两个方法一样，`KeyBERT` 的输入基础仍然是：

- 基础统计模块输出的 `documents`

也就是：

- **单条完整对话 = 一个 document**

### 3.2 先做 token 过滤

项目里会先对每条文档调用：

- `filter_document`

然后再把过滤后的 token 重新拼接成文本。  
这么做是为了让 `KeyBERT` 少看一些明显无意义的噪声词。

### 3.3 如何把文档交给 KeyBERT

这里有一个和很多标准用法不同、但很符合本项目实际需求的细节：

- 每条对话会先变成一行文本
- 再把同一个 split 中所有文档用换行符拼接起来
- 最后把整个拼接结果作为一个大文本，送给 `KeyBERT`

也就是说，本项目当前提取的不是：

- “每条对话各自的关键词”

而是：

- “某个 split 整体的代表性关键词”

这和 `TF-IDF`、`TextRank` 的总体目标是一致的，都是为了看：

- 训练集整体在说什么
- 验证集整体在说什么
- 测试集整体在说什么

### 3.4 关键词长度设置

项目里设置了：

- `keyphrase_ngram_range=(1, 2)`

这表示候选关键词既可以是：

- 单个词
- 也可以是两个词组成的短语

这样做的好处是：

- 不只看单词
- 还能捕捉像“订单号”“售后 服务”这类更完整的局部概念

### 3.5 输出数量

项目中还限制了：

- `top_n = top_k`

也就是只保留前若干个得分最高的关键词，方便汇报和可视化。

## 4. 为什么这个方法适合本项目

### 4.1 它提供了和统计方法不同的视角

如果只看 `TF-IDF` 和 `TextRank`，我们看到的仍然是比较传统的文本统计视角。  
加入 `KeyBERT` 后，就多了一个：

- 语义相似度视角

这能让方法组合更完整。

### 4.2 对高频虚词更不敏感

理论上，像“的、了、是”这类语义弱词，  
即使出现频率高，也不太可能和整段文本的语义中心特别接近。  
因此 `KeyBERT` 比纯频率方法更有机会把它们压下去。

### 4.3 更接近“主题提炼”

在汇报时，如果要说明“除了传统统计方法，我们还尝试了语义级关键词提取”，  
`KeyBERT` 是一个很合适的选择。

## 5. 本项目实现上的注意点

### 5.1 当前实现刻意保持简单

代码里没有强行加入：

- 复杂 batch 控制
- 设备选择
- 底层并行调度

原因不是不能做，而是：

- 这一层如果写得太重，反而破坏代码清晰度

因此当前实现的定位是：

- 先把方法跑通
- 保持代码简洁

### 5.2 依赖安装是可选的

项目里做了 `ImportError` 保护。  
如果本机没装 `KeyBERT`，主流程不会整体崩掉，而是：

- 跳过 `KeyBERT` 这一路

这是一个偏工程化的设计，保证整个统计主流程更稳。

## 6. 局限

### 6.1 成本高于传统方法

相比 `TF-IDF` 和 `TextRank`，  
`KeyBERT` 需要加载语义模型，因此更慢，也更依赖环境。

### 6.2 当前实现是 split 级整体提取

它现在做的是：

- 一个 split 一组关键词

而不是每条对话单独抽。  
这适合当前汇报，但如果未来要做更细粒度分析，可能需要调整。

### 6.3 最终效果依赖底层嵌入模型

`KeyBERT` 自身只是方法框架，效果仍然会受到底层语义模型质量影响。

## 7. 在本项目中的定位

`KeyBERT` 在本项目里最适合的定位是：

- 作为第三种关键词方案
- 补足语义视角
- 与 `TF-IDF`、`TextRank` 做交叉比较

这样在汇报时就可以说：

- 我们既做了传统统计方法
- 也做了图方法
- 还做了基于语义嵌入的方法

方法层次会更完整。
