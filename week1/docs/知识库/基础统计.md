# 基础统计

## 1. 这个方法在项目里做什么

在 `week1/stats` 里，基础统计是所有后续分析的底座。  
它主要回答五个最基本的问题：

1. 一条对话平均有多少轮
2. 一整条对话大概有多少 `token`
3. 单轮 `utterance` 大概有多少 `token`
4. 一整条对话大概有多少词
5. 单轮 `utterance` 大概有多少词

对应代码位置：

- `/Users/bizi/Desktop/邮政实习/week1/stats/basic_analysis.py`

## 2. 原理

严格来说，这一部分不属于复杂算法，而是**统计描述**。  
它的思想很简单：先把每条样本中能量化的长度信息抽出来，再对这些数值做摘要统计。

项目里最终输出的摘要包括：

- `count`
- `mean`
- `median`
- `min`
- `max`

这些量可以快速反映数据集的大致规模和分布范围。

## 3. 本项目里具体怎么做

### 3.1 对话轮数

每条样本里都有 `Dialogue` 字段，它本身就是一个多轮列表。  
因此，对话轮数的计算方式就是：

- 取 `len(Dialogue)`

也就是说：

- 一条样本 = 一条完整对话
- 这条完整对话里有多少个 turn，就记多少轮

### 3.2 token 数统计

项目里 `token` 统计采用的是：

- `tiktoken`
- 编码器使用 `cl100k_base`

这个口径是用户明确指定的。  
同时，在编码之前还做了一步处理：

- **先去掉所有空格**

对应逻辑是：

1. 取出原始文本
2. 用 `remove_spaces(text)` 去掉所有空格
3. 再用 `cl100k_base` 编码
4. 编码结果长度就是 token 数

这样做的原因是：

- 原始数据已经分词，文本里带有很多人为加入的空格
- 如果直接拿带空格文本编码，统计结果会受分词格式影响
- 去掉空格后再编码，更接近“纯内容长度”的比较口径

### 3.3 词数统计

词数统计和 token 数统计不同。  
这里并没有再做分词，而是直接利用数据集本身已经分好词这一事实。

做法是：

- 按空格切分
- 去掉空 token
- 剩余 token 的个数就是词数

所以这里的“词数”本质上是：

- **原始数据里已经分好的词的个数**

### 3.4 整条对话和单轮 utterance 两种粒度

项目里同时保留了两种粒度：

1. **单轮 utterance 粒度**
   - 统计每一轮话术有多长
   - 适合看单轮发言的长度分布

2. **整条对话粒度**
   - 把同一条对话的所有轮次累加
   - 适合看完整客服会话有多长

这两种粒度一起看，才能更完整理解数据：

- 单轮长度反映每次发言的复杂度
- 整体长度反映一整个客服流程的复杂度

## 4. 基础统计为什么还要输出 words 和 documents

虽然这个模块叫“基础统计”，但它还顺手准备了两个后续会复用的重要对象：

### 4.1 `words`

这里保存的是：

- 当前 split 里所有 utterance 的全部词 token 序列

作用是：

- 后续直接做词频统计
- 画普通词云

### 4.2 `documents`

这里保存的是：

- 每条完整对话对应的 token 列表

注意粒度：

- **单条对话 = 一个 document**

这个设计很重要，因为后续：

- `TF-IDF`
- `TextRank`
- `KeyBERT`

都需要建立在某种“文档粒度”之上。  
本项目最终选的是：

- 以完整对话作为一个文档，而不是单轮 utterance

## 5. 输出结果是什么

基础统计模块最后会输出：

- 终端摘要
- `summary.json`

其中 `summary.json` 里保存的是每个数据划分的摘要信息，例如：

- 训练集样本数
- 对话轮数统计
- token 长度统计
- 词数统计

这样做的好处是：

- 结果可重复读取
- 后续文档或汇报可以直接引用

## 6. 优点和局限

### 优点

- 简单稳定
- 解释成本低
- 能快速建立对数据规模和长度分布的直觉
- 为后续高级方法准备了统一输入

### 局限

- 只能告诉我们“多长、多大、多频繁”
- 不能直接告诉我们“主题是什么”
- 无法理解语义

所以它适合做第一层数据理解，但不能替代后续关键词分析。
