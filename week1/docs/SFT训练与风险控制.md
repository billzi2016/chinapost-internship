# SFT 训练与风险控制

## 1. 目标

本文件用于回答两个实际问题：

1. 在当前 `k` 级别中文客服数据上，SFT 应该怎么做更稳
2. 训练过程中应该看哪些指标，避免模型出现“训崩 / crash / collapse / 明显退化”

这里的“风险控制”不是泛泛而谈，而是面向实习项目的可执行方案。

## 2. 当前数据特点决定不能激进训练

结合当前数据现状，至少有三个约束：

### 2.1 数据量不大

当前数据是 `k` 级别，不是百万级别。  
这意味着：

- 很难支撑激进的大模型全参训练
- 很容易过拟合
- 很容易把模型原有通用能力训坏

### 2.2 数据更像客服监督数据，而不是 reasoning 数据

当前数据核心内容是：

- 对话
- summary
- intent

它适合做：

- 客服风格适配
- 领域知识适配
- 工单总结与结构化输出

但它并不天然适合做：

- 高质量思维链监督
- reasoning 行为强化

### 2.3 输出目标很明确

本项目最终希望模型能稳定完成：

1. 中文客服对话
2. 领域问答
3. 工单总结
4. `JSON` 输出

因此训练设计应该围绕“稳定输出”和“任务适配”，而不是盲目追求 reasoning 分数。

## 3. 微调方式建议

## 3.1 优先 LoRA / 参数高效微调

当前阶段不建议全参微调，优先使用：

- `LoRA`
- 或其他参数高效微调方案

原因：

- 算力成本低
- 调参成本低
- 更适合小数据
- 更不容易把底模能力整体训坏

## 3.2 先从小模型跑通，再上更大模型

建议顺序：

1. `Qwen2.5-3B`
2. `Qwen2.5-7B`
3. 视资源情况再考虑 `gpt-oss:20b`

这样做的价值在于：

- 小模型更快暴露数据问题
- 小模型更适合快速试超参数
- 等流程稳定后，再把经验迁移到更大模型

## 3.3 不建议一步到位上超大模型

原因不是“大模型没用”，而是：

- 训练慢
- 验证慢
- debug 成本高
- 实习项目时间窗口里不划算

## 4. 训练时最需要监控的指标

下面这些指标，建议你在报告里明确写成“训练健康度监控项”。

### 4.1 训练 loss

最基础指标。  
需要观察：

- 是否稳定下降
- 是否突然震荡
- 是否在某一阶段开始异常升高

风险信号：

- loss 长时间不降：说明学习率、数据、模板或训练目标有问题
- loss 突然暴涨：可能是学习率过高、梯度不稳定、数据脏样本影响

### 4.2 验证 loss

这是比训练 loss 更重要的指标。  
需要观察：

- 验证集 loss 是否同步改善
- 是否出现明显过拟合

风险信号：

- 训练 loss 继续下降，但验证 loss 开始上升
- 说明模型开始记忆训练集而不是学到泛化能力

### 4.3 结构化输出成功率

这个项目不能只看语言模型的通用 loss，必须看任务指标。  
建议专门统计：

- `JSON` 可解析率
- 字段完整率
- 字段格式正确率

风险信号：

- loss 看起来下降了，但 JSON 经常不闭合
- 字段缺失率很高
- 输出开始变长、变乱、格式漂移

这类情况说明：

- 模型也许“学会了说话”，但没学会“按业务要求输出”

### 4.4 客服回复质量抽样

建议每个 checkpoint 固定抽样同一批验证样本，人工看：

- 回复是否礼貌
- 是否答非所问
- 是否出现明显幻觉
- 是否与邮政场景匹配

原因：

- 纯 loss 指标很难完全反映客服体验
- 实习报告里也需要能展示“人工质检结果”

### 4.5 通用能力退化信号

建议保留一小批通用样本用于观察：

- 常识问答
- 一般对话
- 简单结构化输出

风险信号：

- 客服领域内效果提升了
- 但通用输出明显退化，甚至基础表达能力下降

这通常意味着：

- 灾难性遗忘

## 5. 重点防崩指标：哪些信号最像“模型开始训崩”

如果只能挑最关键的几个，我建议报告里重点写这 5 个：

1. **训练 loss 是否突然暴涨**
2. **验证 loss 是否在训练 loss 继续下降时反向上升**
3. **JSON 可解析率是否下降**
4. **固定验证样本输出是否出现明显格式漂移**
5. **同一 checkpoint 的人工抽样质量是否突然恶化**

这些比只看一个总 loss 更能说明“模型有没有开始坏掉”。

## 6. 为什么会训崩

常见原因包括：

### 6.1 学习率过高

最常见。  
表现：

- loss 抖动大
- 输出开始乱
- 验证波动明显

### 6.2 数据质量不稳定

当前数据本身结构不算特别规整，存在：

- summary / intent / 对话粒度混杂
- 样本风格不统一
- 部分样本噪声较大

如果不清洗干净，容易导致模型学到不稳定模式。

### 6.3 模板设计不一致

如果训练样本的输入输出模板不统一，比如：

- 有的样本像对话
- 有的样本像摘要
- 有的样本像 JSON 生成

但又没有清楚区分任务头，模型会学得很乱。

### 6.4 小数据下训练太久

`k` 级别数据很容易过拟合。  
所以：

- 训练步数过多
- checkpoint 选得太晚

都可能让模型“越训越差”。

## 7. 如何降低训崩概率

### 7.1 保守超参数起步

建议策略：

- 先用保守学习率
- 小步试跑
- 先看 1 个 epoch 内的趋势

不要一开始就把训练拉满。

### 7.2 固定频率保存 checkpoint

建议：

- 按固定 `step` 保存 checkpoint
- 不默认最后一个就是最好

这样一旦后期开始退化，你还能回到更稳的中间点。

### 7.3 每个 checkpoint 做同一套验证

固定验证内容建议包括：

- 验证集 loss
- JSON 可解析率
- 字段完整率
- 客服回复人工抽样

这样不同 checkpoint 才能横向比较。

### 7.4 单独监控 JSON 任务

这个项目里，`JSON` 输出能力是单独的核心能力。  
建议不要把它混在总评价里，而要单独列成一项。

### 7.5 先做 3B，再做 7B

原因很务实：

- 3B 更快出结果
- 更快暴露数据 / 模板问题
- 先把方案跑稳，再迁移到 7B，成功率更高

## 8. 建议在实习报告里写出的训练流程

可以写成如下版本：

1. 先对原始客服数据做清洗、筛选和 EDA
2. 统一训练模板，区分客服对话任务与工单总结 / JSON 任务
3. 先使用 `Qwen2.5-3B` 做 LoRA 微调，验证训练流程和指标体系
4. 固定 step 保存 checkpoint，并持续监控训练 loss、验证 loss、JSON 可解析率和人工抽样效果
5. 在训练稳定后，再迁移到 `Qwen2.5-7B` 获取更强效果
6. 以 `gpt-oss:20b` 作为更大模型基线或补充对照，而非第一优先 SFT 主力

## 9. 最终建议

如果从“实习项目能交付”的角度总结，最稳的训练策略是：

- **主力模型：Qwen2.5-3B / 7B**
- **训练方式：LoRA**
- **关键监控：训练 loss + 验证 loss + JSON 可解析率 + 人工抽样**
- **checkpoint 策略：固定 step 保存，中途选优，不迷信最后一个**

一句话概括：

> 当前项目最怕的不是模型不够大，而是训练不稳定、输出不受控、最后拿不出完整结果。因此，SFT 的首要目标是稳定，而不是盲目追求最强模型。
