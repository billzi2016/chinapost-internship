# Week1: 邮政客服 **SFT** + **RAG** 项目 PRD

## 1. 项目背景

本项目面向邮政客服场景，目标是基于大语言模型构建一个可落地的智能客服系统。系统不仅需要完成用户对话，还需要在对话结束后自动生成结构化工单，从而减少人工客服负担，提升响应效率，并为后续业务流程提供标准化输入。

虽然文件名为 `week1`，但本文档整理的是整个项目的完整任务定义，不仅限于第一周工作。

## 2. 业务目标

核心目标包括两部分：

1. 客服对话
   处理邮政客服常见问题，例如：
   - 物流进度查询
   - EMS / 标快 / 特快咨询
   - 普快 / 普通快递咨询
   - 邮费估算
   - 退费规则咨询
   - 邮政制度与业务规则问答

2. 工单生成
   在客服对话结束后，自动生成工单总结，并输出结构化 `JSON`，例如：
   - 用户 ID
   - 时间
   - 咨询类型
   - 用户诉求
   - 处理结果
   - 是否需要人工跟进

## 3. 业务价值

项目落地后需要能够说明以下业务收益：

- 减少人工客服数量或人工介入频率
- 提升客服响应速度
- 缩短工单填写时间
- 提高回答规范性与一致性
- 提升工单结构化程度，便于后续流转、统计和分析

后续汇报或面试时，不能只讲“做了模型”，还要讲清楚：

- 为什么做这件事
- 解决了什么业务问题
- 最终带来了什么效果

## 4. 业务场景范围

当前场景聚焦邮政客服，覆盖两类主要业务：

- EMS / 标快 / 特快
- 普通快递 / 物流配送

高频问题包括：

- “我的快递到哪了”
- “能不能退费”
- “邮费怎么算”
- “多久能送到”
- “不同服务类型的资费规则是什么”

## 5. 总体方案

整体方案采用 **SFT** + **RAG** 结合的思路：

1. 用 **SFT** 提升模型在邮政客服场景下的对话能力和工单生成能力
2. 用 **RAG** 作为兜底机制，解决幻觉、回答不规范、回答不完整的问题
3. 通过评估体系与工程化部署，形成完整项目闭环

## 6. 系统目标能力

系统需要具备以下能力：

1. 多轮客服对话
2. 邮政领域问答
3. 自动生成工单摘要
4. 输出结构化 `JSON`
5. 在回答不可靠时触发 **RAG** 检索兜底
6. 支持模型效果评估、推理速度评估、资源消耗评估
7. 支持后续部署为 API 或可视化界面

## 7. 项目拆解

整个任务可以拆成四大块：

1. 数据处理与分析
2. **模型选型**与 **SFT**
3. **RAG** 知识库与兜底机制
4. 评估、**量化**、**部署**与总结

## 8. 数据处理与分析

### 8.1 数据来源

优先使用现成客服对话数据集，而不是一开始就全量自建标注。已有数据集中包含多种客服场景，需要从中筛选出与物流、邮政相关的部分。

数据集中可能同时包含：

- 电商客服
- 物流客服
- 其他通用客服对话

本项目只保留物流 / 邮政相关数据。

### 8.2 数据清洗

需要完成基础数据清洗工作，例如：

- 去除空白字符和异常格式
- 清理脏数据
- 统一字段格式
- 检查轮次数异常的对话
- 检查无效总结或缺失字段

### 8.3 **数据筛选**

重点任务是从混合客服数据中筛出物流相关对话。可采用以下方式：

- 正则
- 关键词匹配
- 人工抽样校验

关键词库可围绕以下主题维护：

- 物流
- 邮政
- EMS
- 标快
- 特快
- 邮费
- 运费
- 退费
- 揽收
- 签收
- 派送
- 工单

这一步要形成清晰的**数据筛选**策略，而不是只说“把物流数据拿出来了”。

### 8.4 EDA 与可视化

在正式训练前，需要完成数据集探索分析 `EDA`，至少包括：

- 数据总量
- 物流相关数据占比
- 平均对话轮数
- 对话长度分布
- 高频议题分布
- 总结字段分布

可以补充可视化产出，例如：

- 轮次分布图
- 文本长度分布图
- 主题分布图
- **词云**

**词云** 不是必须，但如果做出来，适合放到汇报、周报、面试材料里，增强表达效果。

## 9. **模型选型**

### 9.1 选型原则

**模型选型**不能只凭“自己熟悉”决定，需要结合：

- 中文能力
- 指令跟随能力
- 多轮对话能力
- `JSON` 输出能力
- 微调可行性
- 推理速度
- 显存占用
- 当前算力环境

### 9.2 候选方向

从会议内容看，中文场景下优先考虑 `Qwen` 系列的小模型路线，例如：

- `Qwen2.5-7B`
- `Qwen2.5-8B`
- 其他同量级中文模型

讨论中提到，部分 `MOE` 模型不适合当前任务直接做 `SFT`，原因包括：

- 训练支持不稳定
- 同步机制复杂
- 更容易出现训练 collapse
- 工程复杂度更高

因此项目初期更适合采用稠密小模型，而不是复杂 `MOE` 路线。

### 9.3 选型结论建议

建议先从 `7B / 8B` 中文模型起步，理由：

- 参数规模适中
- 推理和微调成本可控
- 适合做第一版实验
- 更容易完成从训练到部署的完整闭环

如果后续效果不足，再对比更大模型，但要同时评估：

- 效果提升多少
- 推理速度下降多少
- 算力成本增加多少

## 10. **SFT** 任务设计

### 10.1 **SFT** 的两个核心任务

**SFT** 不是只做一个任务，而是同时覆盖：

1. 客服对话生成
2. 工单总结 / `JSON` 生成

也就是说，模型既要能和用户对话，也要能在对话后产出结构化结果。

### 10.2 微调方式

当前任务不建议全参微调，优先使用参数高效微调方案，例如：

- **LoRA**
- 低秩矩阵微调

原因：

- 当前任务是领域适配，不是从头训练基础能力
- 全参微调算力成本高
- 更容易破坏原始模型通用能力
- 参数高效微调更符合当前项目约束

### 10.3 训练风险

`SFT` 的重点风险包括：

- **灾难性遗忘**
- 训练过度导致原始能力下降
- `JSON` 输出不稳定
- 领域对话有提升，但泛化能力下降

因此训练过程中必须保留验证与对比机制。

### 10.4 训练监控

建议按固定 step 保存 checkpoint，并在小验证集上持续评估，例如：

- 每 30 step 保存一次
- 比较不同 checkpoint 在验证集上的效果
- 选择最优 checkpoint，而不是默认最后一个

必要时可额外做“保能力”验证，例如：

- 使用通用验证集监控退化
- 关注原始能力是否明显下滑

## 11. **工单 JSON** 生成

**工单 JSON** 生成是项目里非常关键的一部分，不应视为附属功能。

目标输出示例字段可包括：

- `user_id`
- `timestamp`
- `service_type`
- `issue_type`
- `user_request`
- `summary`
- `resolution`
- `need_follow_up`

需要重点关注：

- 字段是否完整
- 格式是否稳定
- 能否严格输出合法 `JSON`

如果微调后的主模型 `JSON` 能力不稳定，可以增加一层后处理：

- 先生成自然语言总结
- 再由单独的小模型或规则层转换为 `JSON`

## 12. **RAG** 兜底设计

### 12.1 **RAG** 的定位

**RAG** 不是主流程替代品，而是兜底机制。

适用场景包括：

- 模型回答出现幻觉
- 回答不完整
- 回答不够规范
- 涉及明确制度、规则、收费标准时

### 12.2 **邮政知识库**内容

可纳入**邮政知识库**的内容包括：

- 邮政官方制度
- 邮费规则
- 退费规则
- 配送时效规则
- 不同业务类型说明
- 常见问题文档

### 12.3 **RAG** 流程

建议流程如下：

1. 先由微调模型直接回答
2. 对回答结果做质量判断
3. 若判断结果不佳，则触发 **RAG**
4. 从知识库检索相关内容
5. 基于检索结果生成偏官方、偏规范的答案

### 12.4 **RAG** 触发条件

可设计以下触发逻辑：

- 语义完整性不足
- 规范性不足
- 答案置信度低
- 命中制度类 / 规则类问题

其中“语义完整性”是会议中反复提到的关键点。

## 13. 评估体系

项目评估不能只看单一指标，需要分层设计。

### 13.1 自动评估

可使用的自动指标包括：

- **困惑度**
- 回答准确率
- `JSON` 结构正确率
- 推理速度
- 显存 / 资源消耗

其中 **困惑度** 可用于评估生成流畅性，但不能代表业务正确性。

### 13.2 语义层评估

最终更重要的是：

- **语义一致性**
- 业务正确性
- 表达规范性
- 回答完整性

也就是说，项目重点不是字符串一致，而是语义是否对、是否符合邮政话术和业务逻辑。

### 13.3 人工评估

人工评估建议从以下维度打分：

- 语义是否一致
- 语法是否通顺
- 表达是否规范
- 业务规则是否正确
- 是否满足用户诉求

### 13.4 **语义相似度**辅助评估

会议中提到可以用模型辅助做评估，例如：

- 计算参考答案与模型答案的**语义相似度**
- 设置阈值，例如 `0.65`
- 高于阈值则认为回答基本正确

这类方式不能完全替代人工评估，但能减少人工工作量。

## 14. 工程化与**量化**

### 14.1 工程目标

项目不应停留在“训练完一个模型”，还需要工程闭环，包括：

- 推理服务
- API
- 基本界面
- 部署方式
- 推理性能优化

### 14.2 **量化**

需要比较量化前后的：

- 模型效果损失
- 推理速度提升
- 资源消耗变化

重点不是绝对最优，而是工程可用性。

### 14.3 部署

部署目标可以是：

- 本地推理服务
- API Server
- 简单 Web 界面
- 基于 `Ollama` 的推理封装

如果时间充足，可以让系统从“能跑”提升到“能展示”。

## 15. 算力与资源约束

从当前条件看，项目具备较强 GPU 资源，因此：

- 小模型微调是可行的
- 单卡即可完成大部分实验
- 后续如有必要，可探索多卡

但不建议因为资源充足就一开始追大模型，应优先完成：

1. 可训练
2. 可评估
3. 可部署
4. 可解释

项目最终应至少形成以下产出：

1. 数据分析文档
   包括数据规模、轮次、主题分布、筛选策略、**词云**、样例分析

2. **模型选型**文档
   包括候选模型、对比理由、最终选择、算力约束说明

3. **SFT** 实验记录
   包括训练配置、checkpoint 对比、验证集表现、问题分析

4. **RAG** 方案文档
   包括知识库来源、检索流程、触发逻辑、兜底策略

5. 评估报告
   包括 **困惑度**、**语义一致性**、人工评估、**语义相似度**、推理速度

6. 工程**部署**说明
   包括 API、推理方式、**量化**方案、运行示例

7. 项目总报告 / PPT
   用于周报、汇报、面试复盘

## 17. 推荐执行顺序

建议按下面顺序推进：

1. 做数据 `EDA`
2. 清洗数据并筛出物流 / 邮政相关子集
3. 完成**模型选型**调研
4. 做第一版 **SFT**
5. 做**工单 JSON**输出验证
6. 构建**邮政知识库**并接入 **RAG**
7. 建立评估体系，重点关注 **困惑度**、**语义一致性**、人工评估
8. 做**量化**与推理速度对比
9. 完成 API / 界面 / **部署**
10. 输出文档、周报、总结和面试材料

## 18. 第一周建议任务

如果按周推进，第一周重点应放在两件事：

1. 数据调研
   - 看数据结构
   - 做 `EDA`
   - 清洗文本
   - **数据筛选**物流相关数据
   - 建关键词库 / 正则规则
   - 视情况做可视化和 **词云**

2. 模型调研
   - 列出候选模型
   - 分析中文能力、参数规模、训练可行性
   - 结合算力与工程条件给出**模型选型**首选方案

## 19. 面试与汇报表达重点

后续无论是向导师、公司还是面试官汇报，都应强调以下逻辑：

- 这是一个真实业务场景，不是纯算法题
- 方案不是只有 **SFT**，而是 **SFT + RAG**
- 数据处理、**模型选型**、评估、**部署**形成了完整闭环
- 指标不仅有 **困惑度**，更重视 **语义一致性**
- **词云**、数据分布、案例分析等材料能帮助表达项目理解深度
- 最终目标不是“训练出一个模型”，而是“做出一个可解释、可评估、可部署的邮政客服系统”
