# TF-IDF 方法说明

## 1. 方法定位

`TF-IDF` 是本项目关键词分析里的传统统计基线。  
它的作用是：

- 找出在某些对话里比较重要
- 但又不是“所有对话里都到处出现”的词

对应代码位置：

- `/Users/bizi/Desktop/邮政实习/week1/stats/tfidf_analysis.py`

## 2. 原理

`TF-IDF` 由两部分组成：

### 2.1 TF：Term Frequency

`TF` 表示一个词在当前文档里出现得有多频繁。

本项目中的公式思想是：

- 某个词在当前文档里出现次数越多
- 它在当前文档中的重要性就越高

代码里使用的是归一化后的频率：

- `tf = count / total_terms`

也就是：

- 当前词在该文档中的出现次数
- 除以该文档总词数

### 2.2 IDF：Inverse Document Frequency

`IDF` 表示一个词是不是在很多文档里都常见。

如果一个词：

- 几乎所有文档里都有

那它通常不适合作为“区分主题”的关键词。  
因此，`IDF` 会压低这种词的权重。

项目里用的是平滑版公式：

- `idf = log((1 + N) / (1 + df)) + 1`

其中：

- `N` 是文档总数
- `df` 是包含该词的文档数

直觉上就是：

- 出现在越多文档里的词，`idf` 越低
- 只出现在一部分文档里的词，`idf` 越高

### 2.3 TF-IDF 分数

最终分数是：

- `tf * idf`

也就是：

- 当前文档里很常见
- 但在整个语料里又不至于泛滥

的词，会得到更高权重。

## 3. 本项目里具体怎么做

### 3.1 文档粒度是什么

本项目里最关键的实现约定是：

- **单条完整对话 = 一个 document**

这不是按单轮 utterance 算的。  
原因是：

- 主题通常体现在整条对话中
- 单轮话语太短，容易噪声太大

因此，基础统计模块先把每条完整对话整理成一个 token 列表，再交给 `TF-IDF` 使用。

### 3.2 先做文档过滤

在真正计算之前，代码会先通过 `keyword_utils.filter_document` 对每条文档做过滤。  
这一层的作用是尽量去掉：

- 太弱的噪声词
- 不适合做关键词的高频虚词
- 不满足项目过滤规则的 token

这一步很重要，因为如果不先过滤，中文里大量结构助词、口语套话、客服固定表达会把结果冲淡。

当前实现里的过滤不是抽象概念，而是有明确规则的。  
例如，现有 `STOPWORDS` 和 `SINGLE_CHAR_STOPWORDS` 已经排除了：

- `的`、`了`、`是`
- `亲`、`亲亲`、`亲爱`
- `您`、`你好`、`您好`
- `请问`、`麻烦`、`稍等`
- `订单`、`申请`、`查询`、`订单号`

此外，当前实现还会继续过滤：

- 单字 token
- 纯数字 token
- 方括号包裹的特殊 token

这意味着本项目的 `TF-IDF` 不是“裸跑”版本，而是先经过一层明显偏工程化的拒绝词和 token 清洗。

### 3.3 统计文档频率和词频

项目实现里会做两类统计：

1. 每篇文档内部的词频 `Counter`
2. 每个词出现在多少篇文档中，也就是 `document_frequency`

然后再对每篇文档中的每个词计算：

- `tf`
- `idf`
- `tf * idf`

### 3.4 分数如何汇总

这里有一个很关键的实现细节：  
本项目不是只看单篇文档的分数，而是把词在所有文档中的 `tf-idf` 分数**累加**起来，得到全局汇总分数。

也就是说，最终得到的是：

- 一个词在整个 split 中累积出来的代表性得分

这样做之后，输出的就不再是“某一篇文档的关键词”，而是：

- 某个数据集划分整体上最有代表性的关键词

### 3.5 额外的文档频率过滤

项目里还做了一层额外限制：

- 文档频率过低的词不要
- 文档频率过高的词也不要

原因是：

- `df` 太低：可能只是偶然词，代表性不够
- `df` 太高：说明几乎所有对话都在说它，区分能力太弱

这一步是在项目环境中提升结果可读性的关键，不是所有教材版 `TF-IDF` 都会默认这么做。

当前实现中，这两个阈值是明确写死的：

- `MIN_DOC_FREQ = 5`
- `MAX_DOC_FREQ_RATIO = 0.45`

也就是说：

- 少于 5 篇文档出现的词，不进入结果
- 出现在 45% 以上文档里的词，也不进入结果

这套限制的目的很明确：  
既不让偶然词进来，也不让“虽然常见但没有主题区分能力”的泛化词继续占据结果前列。

## 4. 输出结果是什么

本项目会为每个 split 和全量语料输出：

- `tfidf_keywords.json`
- `tfidf_keywords.txt`

每个关键词记录至少包含：

- `token`
- `score`
- `document_frequency`

因此后续既能做人读，也能继续做程序处理和绘图。

## 5. 适合本项目的原因

`TF-IDF` 适合这个项目，主要因为它有三点价值：

### 5.1 可解释性强

它不是黑盒。  
如果老师或带教问“为什么这个词分数高”，可以直接解释：

- 因为它在部分对话里频繁出现
- 但又不是所有对话都一样高频

### 5.2 能作为稳定基线

即使后面用了 `TextRank`、`KeyBERT` 甚至 `LLM`，  
也仍然需要一个传统统计方法做参照。  
`TF-IDF` 很适合担任这个角色。

### 5.3 对数据量要求不高

相比语义模型方法，`TF-IDF` 不依赖大模型，也不需要额外训练。  
对于 `k` 级别数据，它很容易直接跑通。

## 6. 局限

### 6.1 仍然是词袋思想

它本质上还是在“数数”，不是在理解语义。  
因此：

- 同义词不会自动合并
- 上下文含义不会真正被理解

### 6.2 仍然依赖过滤质量

如果停用词和文档过滤做得不好，  
它还是可能把很多不重要但常见的词顶上来。

### 6.3 不擅长短文本语义凝聚

客服对话里很多表达很碎，  
仅靠 `TF-IDF` 并不一定能抓住真正的业务意图。

因此，本项目把它定位为：

- **重要的统计基线**
- **不是唯一的高级关键词方案**
