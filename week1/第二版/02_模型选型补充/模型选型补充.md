# 模型选型补充：训练主线与 JSON 结构化节点分工

## 1. 补充目的

第二版反馈要求模型选型不能只停留在“哪个模型能力强”，还需要从训练效率、量化后内存占用、CPU/本地推理速度、中文语义理解、长对话上下文、JSON 结构化输出准确率、商用合规性等角度补充。

结合本项目的实际任务，模型路线不建议写成“只选一个模型解决所有问题”，而应写成分工式流程：

- `Qwen2.5` 系列负责中文邮政客服任务的 SFT 主线。
- `gpt-oss:20b` 作为结构化 JSON 生成、格式修复、字段校验的工具节点或流程节点。

这样设计的核心原因是：中文客服 SFT 与 JSON 结构化输出并不是同一个最优解问题。前者更看重中文语义、客服语气、小数据微调稳定性和训练成本；后者更看重格式服从、schema 对齐、字段完整性和后处理成本。

## 2. 本机 Ollama 模型情况

本机已安装多个候选模型，和当前任务最相关的是：

| 模型 | 本机文件大小 | 参数/结构 | 上下文长度 | 量化方式 | 许可证/使用限制 | 当前定位 |
|---|---:|---:|---:|---|---|---|
| `qwen2.5:3b` | 1.9GB | 3.1B dense | 32K | Q4_K_M | Qwen Research License | 快速实验、流程验证、低成本 SFT 起点 |
| `qwen2.5:7b` | 4.7GB | 7.6B dense | 32K | Q4_K_M | Apache 2.0 | 中文客服 SFT 主力候选 |
| `qwen2.5:14b` | 9.0GB | 14.8B dense | 32K | Q4_K_M | Apache 2.0 | 能力上探、较强中文基线 |
| `gpt-oss:20b` | 13GB | 20.9B MoE，约 3.6B active | 128K | MXFP4 | Apache 2.0 | JSON 结构化生成/校验节点，强本地对照 |

从工程角度看，`qwen2.5:3b` 和 `qwen2.5:7b` 的模型文件明显更小，更适合作为高频实验对象；`qwen2.5:14b` 能力更强但训练与推理成本更高；`gpt-oss:20b` 虽然总参数更大，但 MoE 激活参数较少，并且官方和本机均显示其具备较强的本地部署可行性。

## 3. 本机单条推理测试

测试 prompt：

```text
请判断下面用户问题属于哪类邮政客服意图，只输出JSON：用户说“寄EMS到上海大概要几天，多少钱？”字段为intent、reason、confidence。
```

测试结果：

| 模型 | 单条推理 wall time | 输出格式观察 | 语义观察 |
|---|---:|---|---|
| `qwen2.5:3b` | 10.6 秒 | 输出了 JSON，但包在 Markdown 代码块中 | 能识别为物流/EMS 相关，理由基本正确 |
| `qwen2.5:7b` | 11.5 秒 | 输出了 JSON，但包在 Markdown 代码块中，`confidence` 为字符串 | 意图更贴近“时效和费用查询” |
| `qwen2.5:14b` | 13.5 秒 | 输出裸 JSON，格式较稳定 | 理由完整，但意图标签略泛 |
| `gpt-oss:20b` | 5.6 秒 | CLI 中展示 thinking，最终答案为裸 JSON | 意图标签最细，字段完整度较好 |

这组测试不是严格 benchmark，只是本机同题快速验证。它能说明两个方向：

1. `Qwen2.5` 系列对中文邮政问题理解稳定，适合作为 SFT 主线。
2. `gpt-oss:20b` 在同题下最终 JSON 字段更完整，意图标签更细，适合放在结构化输出节点中。

需要注意的是，`gpt-oss:20b` 在 Ollama CLI 中会显示 thinking 过程。正式流程中不应直接把 CLI 全量文本当作业务输出，而应使用 API 调用、响应字段分离、最终 JSON 提取或 schema 校验方式，只保留最终结构化结果。

## 4. 多维度选型对比

| 维度 | Qwen2.5-3B | Qwen2.5-7B | Qwen2.5-14B | gpt-oss:20b |
|---|---|---|---|---|
| 训练效率 | 最高，适合快速试错 | 较高，适合主线 SFT | 中等，成本明显上升 | 不建议作为当前中文客服 SFT 主线 |
| 量化后本机文件大小 | 1.9GB | 4.7GB | 9.0GB | 13GB |
| 本地推理速度 | 快，但格式稳定性一般 | 较快，语义更稳 | 更稳但更重 | 本次测试最快，适合独立工具节点 |
| 中文语义理解 | 可用 | 稳定 | 更强 | 可用，但官方定位更偏通用推理和 agent |
| 长对话上下文 | 本机 32K | 本机 32K | 本机 32K | 本机 128K |
| JSON 结构化输出 | 需要 prompt/后处理约束 | 需要 prompt/后处理约束 | 较稳 | 最适合做 JSON 生成/修复/校验 |
| 商用合规性 | 3B 为 Qwen Research License，需要谨慎 | Apache 2.0 | Apache 2.0 | Apache 2.0 |
| 当前项目定位 | 快速实验 | SFT 主力 | 能力上探 | 结构化节点/强对照 |

## 5. 为什么 SFT 主线仍选 Qwen2.5

邮政客服任务的主体是中文多轮对话，核心不是复杂推理，而是：

- 理解中文口语表达、省略表达和客服上下文；
- 学会邮政业务场景中的固定话术和回复风格；
- 在小规模数据上稳定收敛；
- 支持 LoRA/PEFT 等低成本微调；
- 能用较低成本反复实验。

`Qwen2.5` 官方资料明确强调中文等多语言能力、指令跟随、长文本、结构化数据理解和 JSON 输出能力。对于本项目来说，它的优势不是单点最强，而是中文、训练、部署、社区生态之间比较均衡。

具体路线建议：

- `qwen2.5:3b` 用于早期快速验证：字段设计、数据格式、训练脚本、loss 下降趋势、过拟合风险。
- `qwen2.5:7b` 用作主力 SFT：在成本可控的前提下获得更好的中文客服表达和意图理解。
- `qwen2.5:14b` 用作能力上探：如果 7B 表达能力不够，再作为更强候选进入对比。

## 6. 为什么 gpt-oss:20b 不作为 SFT 主线，而作为 JSON 节点

`gpt-oss:20b` 的优势非常明确：

- 官方定位支持 Structured Outputs、工具调用和 agentic workflow；
- 本机模型为 MXFP4 量化，文件大小 13GB，官方说明可在 16GB 内存级别运行；
- 上下文长度 128K，适合长对话汇总、批量字段抽取和复杂 schema 处理；
- 本次同题测试中最终 JSON 字段完整、标签细、格式更适合作为结构化结果。

但它不适合作为当前中文客服 SFT 主线，原因也很清楚：

- 官方预训练数据重点偏英语、STEM、代码和通用知识；
- 模型定位更偏 reasoning、agent 和工具使用；
- MoE 架构用于训练和微调时工程复杂度更高；
- 当前项目数据规模是 k 级别，优先目标是稳定中文客服闭环，而不是训练一个更复杂的通用推理模型。

因此更合理的定位是：不让 `gpt-oss:20b` 承担主模型训练，而是让它承担结构化输出流程中的关键节点。

## 7. 推荐落地流程

推荐流程如下：

1. 数据侧先整理为标准 SFT 样本，包含用户问题、客服回复、总结、意图、业务字段。
2. 使用 `qwen2.5:3b` 做快速 SFT 试跑，验证数据格式和训练脚本。
3. 使用 `qwen2.5:7b` 做正式中文客服 SFT，得到主力客服回复模型。
4. 在推理流程后增加 `gpt-oss:20b` 结构化节点，用于：
   - 将对话或模型回复转为固定 JSON；
   - 修复不合法 JSON；
   - 补齐缺失字段；
   - 对字段类型和枚举值做校验；
   - 对长对话做结构化摘要。
5. 对最终 JSON 再做程序校验，包括 `json.loads`、必填字段、字段类型、枚举值和置信度范围。

可以把最终系统写成：

```text
用户问题/多轮对话
  -> Qwen2.5 SFT 客服模型：负责中文客服理解与回复
  -> gpt-oss:20b JSON 工具节点：负责结构化字段生成、修复和校验
  -> 程序侧 schema 校验与落库
```

这种分工比单模型方案更稳。`Qwen2.5` 做它最擅长的中文客服 SFT，`gpt-oss:20b` 做它最擅长的结构化输出和长上下文处理，程序侧再兜底校验格式。

## 8. 最终选型结论

当前阶段推荐：

- 主线训练模型：`qwen2.5:7b`
- 快速实验模型：`qwen2.5:3b`
- 能力上探模型：`qwen2.5:14b`
- JSON 结构化工具节点：`gpt-oss:20b`

报告中可以明确写成：

> 本项目不把 `gpt-oss:20b` 作为中文客服 SFT 主模型，而是将其作为结构化 JSON 生成与校验节点。主模型训练采用 `Qwen2.5` 系列，优先使用 `qwen2.5:7b`，必要时用 `qwen2.5:3b` 快速验证、`qwen2.5:14b` 做能力上探。这样既能利用 Qwen2.5 的中文客服与小数据微调优势，也能利用 gpt-oss:20b 在 JSON 结构化输出、长上下文和工具化流程中的优势。
