# 01/04 合并分析方案：regex 弱标签 + gpt-oss:120b 业务分类

## 1. 为什么 01 和 04 要合并处理

反馈 01 关注的是 `gpt-oss:20b` 二分类结果是否可靠，有没有量化评估和边界 case。  
反馈 04 关注的是可视化时能否看出不同业务簇，并把点分成费率价格类、时限时效类、流程规则类、禁限寄类、国际业务类等。

这两个问题本质上是同一条分析链：

1. 先判断样本是否真的属于邮政 / EMS / 寄递业务。
2. 再对确认为相关的样本细分业务类型。
3. 最后把业务类型作为可视化标签，用于解释 embedding 空间中的簇。

因此第二版应把 01 和 04 合并成一个“标签质量评估 + 业务主题细分”流程。

## 2. 当前二分类结果概况

`week2/data/llm_filter/postal_filter_results.json` 中的 `gpt-oss:20b` 二分类结果为：

| split | 总样本数 | 判为邮政/快递相关 | 判为其他 | 相关占比 |
|---|---:|---:|---:|---:|
| train | 9101 | 5376 | 3725 | 59.07% |
| val | 800 | 459 | 341 | 57.37% |
| test | 800 | 486 | 314 | 60.75% |

这个比例说明：当前二分类标签覆盖面偏宽，更像“快递 / 物流 / 配送 / 订单流转相关”，不完全等于“严格邮政 / EMS / 邮政寄递业务”。

## 3. regex 弱标签设计

regex 的作用不是替代大模型，而是提供一套可解释、可复现的弱标签，用来评估二分类结果和定位边界样本。

建议分成两层：

### 3.1 严格邮政 / EMS 相关关键词

用于判断是否属于更严格的邮政业务：

```text
邮政|中国邮政|EMS|邮局|邮政快递|邮政包裹|特快专递|挂号信|平邮|邮政编码|邮政网点|邮政营业厅
```

这层标签主要解决一个问题：把普通电商配送、京东物流、商家退货、订单签收异常和真正邮政业务区分开。

### 3.2 业务子类关键词

用于给样本打业务类型弱标签：

| 类别 | 关键词示例 |
|---|---|
| 费率价格类 | 运费、邮费、多少钱、价格、费用、收费、保价、报价、首重、续重 |
| 时限时效类 | 几天、多久、什么时候到、时效、预计、送达、到达、延误、超时、派送、物流、快递 |
| 流程规则类 | 怎么寄、怎么下单、修改地址、地址写错、取消、退回、签收、揽收、取件、上门、站点、网点、流程、规则 |
| 禁限寄类 | 禁寄、限寄、不能寄、可以寄吗、违禁、液体、电池、药品、食品、化妆品、危险品 |
| 国际业务类 | 国际、国外、海外、海关、清关、报关、跨境、进口、出口 |

## 4. regex 初步统计

基于 `FinalSumm`、`QA.QASumm` 和 `QA.intent` 做初步关键词匹配，得到以下弱标签统计：

| 类别 | 全量命中数 | train | val | test |
|---|---:|---:|---:|---:|
| 费率价格类 | 1331 | 1133 | 107 | 91 |
| 时限时效类 | 3198 | 2725 | 240 | 233 |
| 流程规则类 | 4256 | 3603 | 332 | 321 |
| 禁限寄类 | 57 | 49 | 5 | 3 |
| 国际业务类 | 77 | 68 | 4 | 5 |

这个结果说明：

- 数据里最多的是流程规则类和时限时效类。
- 费率价格类有一定规模。
- 禁限寄类和国际业务类在当前 CSDS 样本中很少，需要后续人工或大模型补充确认。
- 如果只靠原二分类标签，会把大量普通电商物流和订单售后流程也纳入“邮政相关”，因此必须增加严格邮政判断。

## 5. gpt-oss:120b 的定位

`gpt-oss:120b` 不建议用于全量无差别逐条跑完所有样本，原因是成本高、耗时长。  
更合理的用法是作为高质量离线标注节点：

1. 对 regex 和 `gpt-oss:20b` 不一致的样本做仲裁。
2. 对被 20B 判为相关、但 regex 严格邮政关键词未命中的样本做边界 case 判断。
3. 对可视化聚类中的代表样本做业务类命名。
4. 对少量高价值样本生成更细的业务标签。

建议让 120B 输出固定字段：

```json
{
  "strict_postal": true,
  "category": "时限时效类",
  "reason": "用户询问 EMS 寄达时间",
  "confidence": 0.95
}
```

其中：

- `strict_postal` 判断是否明确属于中国邮政 / EMS / 邮政寄递业务。
- `category` 只允许取固定枚举：费率价格类、时限时效类、流程规则类、禁限寄类、国际业务类、泛物流配送类、非邮政相关。
- `reason` 用一句话说明判断依据。
- `confidence` 用于后续人工复核排序。

## 6. 120B 抽样试跑观察

从 `gpt-oss:20b` 判为邮政相关的 train 样本中抽取前 12 条，让 `gpt-oss:120b` 进一步判断是否属于严格邮政业务。

试跑结果很关键：

- 12 条样本全部被判为 `strict_postal=false`。
- 多数样本属于电商订单、退货、取消订单、修改地址、京东配送、商品安装、发票等场景。
- 其中一部分可以归为 `泛物流配送类` 或 `流程规则类`，但不能算严格邮政 / EMS 业务。

代表样本：

| index | 原二分类 | 120B 判断 | 120B 类别 | 说明 |
|---:|---|---|---|---|
| 0 | 邮政相关 | `strict_postal=false` | 流程规则类 | 地址修改、优惠券退回，属于电商订单流程 |
| 7 | 邮政相关 | `strict_postal=false` | 时限时效类 | 取消订单、配送中、退款周期，属于售后时效 |
| 9 | 邮政相关 | `strict_postal=false` | 非邮政相关 | 发票类型修改，与邮政寄递无关 |
| 14 | 邮政相关 | `strict_postal=false` | 泛物流配送类 | 明确是京东特色配送和修改地址 |
| 17 | 邮政相关 | `strict_postal=false` | 非邮政相关 | 商品参数和安装服务咨询 |

这说明原来的 `gpt-oss:20b` 二分类 prompt 更接近“泛物流/配送相关”判别，召回较高，但精度不足。第二版报告里应该明确这一点，而不是直接把 20B 的结果当成严格邮政标签。

## 7. 最终推荐流程

推荐把 01/04 做成下面的流程：

```text
CSDS 原始样本
  -> 提取 FinalSumm / QASumm / intent / Dialogue
  -> regex 弱标签
       - strict_postal_regex
       - business_category_regex
  -> 对比 gpt-oss:20b 二分类标签
       - 一致样本：作为较可信标签
       - 不一致样本：进入边界 case 池
  -> gpt-oss:120b 离线仲裁
       - 判断 strict_postal
       - 细分业务类
       - 给出 reason / confidence
  -> 业务类标签回填到可视化
       - 费率价格类
       - 时限时效类
       - 流程规则类
       - 禁限寄类
       - 国际业务类
       - 泛物流配送类
       - 非邮政相关
```

## 8. 报告中应写入的结论

第二版报告建议这样写：

> 第一版中 `gpt-oss:20b` 二分类结果更接近“快递/物流/配送/订单流转相关”标签，而不是严格意义上的“邮政/EMS 业务”标签。为评估标签质量，第二版增加了基于 `FinalSumm`、`QASumm` 和 `intent` 的 regex 弱标签，并进一步使用 `gpt-oss:120b` 对边界样本做离线仲裁。初步抽样显示，部分被 20B 判为相关的样本实际属于电商配送、订单售后、退货退款、发票或商品安装，并不属于严格邮政业务。因此后续可视化不应只使用二分类颜色，而应引入业务子类标签和边界样本标记。

可视化部分建议写：

> 在可视化上，第二版将不再只区分“相关/不相关”，而是将邮政相关样本进一步标注为费率价格类、时限时效类、流程规则类、禁限寄类、国际业务类，同时单独保留泛物流配送类和非邮政相关类。这样可以更清楚地解释 embedding 空间中不同簇的业务含义，也能暴露原二分类标签中的边界 case。
