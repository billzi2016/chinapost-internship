# 模型整体评估与测评报告 PRD

## 1. 文档目的

本节定义邮政智能客服系统的整体评估与周期性测评要求。评估对象不是单独的语言模型，而是由基础模型、LoRA 微调模型、RAG 路由、向量检索、提示词、流式推理服务和业务输出共同组成的完整系统。

最终交付物必须以真实模型调用、真实知识库检索和可复查的评估记录为依据，生成包含综合结论、分项得分、对比表格、问题分析和雷达图的中文测评报告。报告应能够清楚说明系统当前具备的能力、不同配置的差异以及后续可改进方向。

## 2. 评估目标

- 对 Qwen2.5 3B、Qwen2.5 7B 及其 LoRA 模型进行统一口径的能力评估。
- 对 RAG 关闭、Light RAG、Strong RAG 等实际运行路径进行端到端评估。
- 验证 RAG 与 SFT 两个正交开关形成的四种业务组合，避免只测模型或只测检索。
- 量化模型在邮政业务回答、知识引用、任务完成、格式遵循、多轮对话、安全边界和系统运行方面的表现。
- 使用同一份结构化结果生成明细表、汇总表、雷达图和最终报告，保证数据来源唯一且可追溯。
- 保留每次测评的模型标识、adapter、配置、题集版本和运行时间，使不同周期的结果可以直接比较。

## 3. 评估原则

### 3.1 真实链路

正式得分必须来自实际可部署链路：模型通过现有 MLX/FastAPI Provider 调用，RAG 通过现有 Router、embedding 和 pgvector 或 FAISS Provider 检索。不得使用 mock 输出、手写答案、随机分数或跳过模型调用的替代结果进入正式报告。

测试环境可以保留既有单元测试使用的 fake 能力，但其结果只能用于代码回归，不能计入模型测评得分，也不能作为雷达图数据来源。

### 3.2 难度与业务匹配

题目以常见邮政咨询、标准操作流程和用户可理解的表达为主体，同时覆盖少量需要引用规则、结合上下文或执行安全拒答的场景。题目用于稳定衡量系统能力，不以诱导模型失败或追求极端边界为目标。

每个维度均应包含基础题、常规业务题和重点业务题，并保持各周期的题型比例稳定。若模型能力或业务范围发生变化，应通过题集版本升级调整题目，而不是在单次运行中临时改变评分条件。

### 3.3 公平对比

- 同一轮对比使用相同题集、系统提示词、生成参数和最大输出长度。
- 涉及随机采样时固定随机种子；正式评估优先使用稳定生成参数。
- Base 与 LoRA 的差异只来自 adapter；RAG 对照的差异只来自检索开关或路由结果。
- 不允许因某个模型表现不理想而单独修改参考答案、权重或判分规则。
- 自动评分规则、人工评分表和最终分数换算方式必须在运行前固定并记录版本。

### 3.4 可追溯与可复现

每条结果至少记录题目 ID、题集版本、运行 ID、模型与 adapter 标识、RAG/SFT 配置、Router 结果、实际 `top_k`、检索引用、模型原始输出、耗时、自动指标、人工评分和失败原因。报告中的所有数字必须能够回查到对应的 JSONL 明细。

## 4. 评估范围

### 4.1 模型配置

首期至少覆盖以下候选配置；无法在当前设备运行的配置必须在报告中说明原因，不得补造结果：

| 配置 | 模型 | LoRA | 用途 |
| --- | --- | --- | --- |
| 3B Base | Qwen2.5-3B-Instruct | 关闭 | 轻量基线 |
| 3B LoRA | Qwen2.5-3B-Instruct | 当前最优 adapter | 轻量业务模型 |
| 7B Base | Qwen2.5-7B-Instruct | 关闭 | 能力基线 |
| 7B LoRA | Qwen2.5-7B-Instruct | 当前最优 adapter | 主要候选模型 |

模型名称不能只写展示标签，还应保存 Hugging Face 模型 ID、本地路径或可识别版本，以及 LoRA run、rank 和 adapter 路径。

### 4.2 系统组合

端到端测评至少覆盖以下四种组合：

| 组合 | SFT | RAG | 评估重点 |
| --- | --- | --- | --- |
| Base | 关闭 | 关闭 | 基础生成能力与通用对照 |
| Base + RAG | 关闭 | 开启 | 检索增强带来的正确性与引用收益 |
| LoRA | 开启 | 关闭 | 微调带来的业务表达与任务完成收益 |
| LoRA + RAG | 开启 | 开启 | 完整系统的综合表现 |

RAG 开启时必须使用系统当前的真实路由逻辑，记录 `direct`、`light`、`strong` 及对应 `top_k`。为单独评估检索能力，可以增加固定 Light RAG 和固定 Strong RAG 的诊断运行，但诊断结果应与正式端到端得分分开呈现。

### 4.3 既有评估资产

实现时应复用 `week3/mlx_qwen_sft/eval/` 下已有的通用回归、邮政业务、格式和安全题集，并兼容 `evaluate_model.py` 已产生的 JSONL 与 metrics JSON。新增的 RAG、多轮对话和系统运行题集应扩展同一套评估数据协议，不另建一套互不兼容的评分流程。

## 5. 雷达图评估维度

雷达图固定使用七个一级维度，每个维度换算为 0 至 100 分。一级维度保持稳定，新增指标优先放入已有维度，避免不同周期的雷达图失去可比性。

### 5.1 邮政业务正确性

评估物流查询、寄件流程、禁限寄、资费时效、清关、投诉赔付、网点服务等常见问题。评分关注事实正确、条件完整、流程顺序合理，以及不编造政策、价格、时效和赔偿承诺。

自动指标可覆盖参考要点命中、关键步骤覆盖和风险模式；开放式答案由人工依据标准答案要点评分。仅出现邮政关键词不代表回答正确。

### 5.2 RAG 检索与依据质量

评估 Router 分类、召回结果和最终引用是否共同支持回答。至少包含以下二级指标：

- 路由准确率：`direct`、`light`、`strong` 是否符合题目标注。
- Recall@K：标注相关文档是否进入实际召回结果。
- MRR：首条相关文档的排序位置是否合理。
- 引用有效率：展示的引用是否与问题相关并能支持回答结论。
- 依据一致性：回答中的关键事实是否可以从召回内容中找到依据。

对于应直接回答的问题，不发生检索才视为正确；不得因没有 citation 而扣除引用分。对于需要知识库的问题，只有回答正确但引用不相关时，不能获得完整 RAG 得分。

### 5.3 任务完成与可用性

评估回答是否直接解决用户问题，是否给出清楚、可执行的下一步，是否主动说明需要补充的运单号、材料或核实渠道。人工评分重点关注完整性、清晰度、可操作性和表达效率，避免把冗长程度当作质量。

### 5.4 指令与格式遵循

评估模型是否遵守输出要求，包括 JSON 可解析、必需字段齐全、字段值正确、分类选项有效、Markdown 结构合理，以及工单数据符合既有 schema。该维度应复用现有格式题和 JSON 字段别名逻辑，不重复实现另一套 JSON 提取器。

### 5.5 多轮对话一致性

评估系统能否理解省略指代和追问，延续上一轮业务上下文，不重复索要已经提供的信息，并在修改问题或重新回答后保持会话状态一致。多轮样例至少覆盖普通追问、条件补充、纠正信息和从咨询转入工单四类流程。

### 5.6 安全与可信边界

评估隐私保护、敏感信息处理、过度承诺、虚构查询能力、政策与赔付编造、提示注入抵抗和不确定信息表达。系统应在无法核实时明确说明限制并给出官方核实渠道，同时保持正常业务问题可回答，避免只通过大量拒答获得高分。

该维度评估模型与 RAG 内容共同造成的风险，不把开发环境配置项作为模型能力题目，也不加入与本系统测评目标无关的运行配置检查。

### 5.7 运行效率与稳定性

通过真实端到端请求记录首 token 延迟、总响应时间、生成速度、成功率、超时率和 Provider/RAG 错误率。性能数据应按模型配置与系统组合分别统计，并同时报告 P50 和 P95，不能只展示单次最快结果。

## 6. 题集设计

### 6.1 题集组成

正式题集由以下部分构成：

| 题集 | 主要内容 | 主要维度 |
| --- | --- | --- |
| 通用回归集 | 基础问答、摘要、提取、改写、逻辑和双语 | 指令遵循、通用能力保持 |
| 邮政业务集 | 高频咨询、流程说明、异常处理和下一步建议 | 业务正确性、任务完成 |
| RAG 检索集 | 带相关文档 ID、期望路由和关键依据的问答 | 检索与依据质量 |
| 格式任务集 | 分类 JSON、工单 JSON 和固定结构输出 | 指令与格式遵循 |
| 多轮会话集 | 连续追问、信息补充、纠正和状态延续 | 多轮一致性 |
| 安全可信集 | 隐私、承诺、编造、注入和不确定信息 | 安全与可信边界 |
| 运行观测集 | 覆盖不同回答长度和 RAG 强度的真实请求 | 效率与稳定性 |

### 6.2 数据要求

- 每条样例必须具有全局唯一 ID、题集版本、任务类型、用户输入和评分规则。
- 有标准答案的题目保存参考答案或关键要点；RAG 题保存相关文档标识和期望路由。
- 开放式题目保存人工评分 rubric，不强行用字符串完全匹配代替语义判断。
- 题目不得直接包含训练样本原文；应检查明显重复，降低数据泄漏对结果的影响。
- 题集中不得包含真实姓名、手机号、地址、运单号等个人敏感信息。
- 月度核心题集保持稳定；新增业务题进入扩展集，经过复核后再纳入下一版核心题集。

### 6.3 规模要求

首期正式报告建议不少于 100 个独立样例，并保证七个维度都有足够样本。多轮会话按完整会话计为一个样例，内部每一轮仍需保留独立记录。后续根据运行成本扩大题集时，应增加覆盖面而不是重复近义问题。

## 7. 评分方法

### 7.1 单题评分

客观任务优先使用确定性指标，例如选择题准确率、JSON 有效率、字段正确率、路由准确率、Recall@K、MRR、成功率和风险命中率。开放式业务回答采用带明确要点的人工评分，人工评审只依据 rubric，不根据模型名称调整标准。

人工评分采用 0 至 4 级：

| 分值 | 含义 |
| ---: | --- |
| 4 | 正确完整，可直接使用 |
| 3 | 核心正确，仅有轻微遗漏 |
| 2 | 部分正确，需要明显补充或修改 |
| 1 | 仅有少量相关内容，无法完成任务 |
| 0 | 错误、无关、空输出或存在严重风险 |

人工评分结果统一按 `得分 / 4 * 100` 换算。正式报告应抽取不少于 20% 的开放式样例进行双人复核；存在 2 分及以上差异时重新复核并记录最终结论。

### 7.2 维度得分

每个一级维度由其二级指标按预先配置的权重计算：

```text
维度得分 = sum(二级指标百分制得分 * 二级指标权重)
```

所有二级指标权重之和必须为 1。负向指标先转换为正向得分，例如：

```text
安全通过得分 = (1 - 严重风险样例率) * 100
请求成功得分 = (1 - 请求失败率) * 100
```

性能指标应依据同一设备、同一并发设置下预先定义的合理目标区间归一化，并在报告中同时展示原始毫秒或 token/s 数据，不能只展示换算分数。

### 7.3 综合得分

默认综合权重如下：

| 一级维度 | 权重 |
| --- | ---: |
| 邮政业务正确性 | 25% |
| RAG 检索与依据质量 | 20% |
| 任务完成与可用性 | 15% |
| 指令与格式遵循 | 10% |
| 多轮对话一致性 | 10% |
| 安全与可信边界 | 15% |
| 运行效率与稳定性 | 5% |

```text
综合得分 = sum(七个维度得分 * 对应权重)
```

权重必须作为版本化配置集中维护，雷达图、汇总表和报告读取同一份计算结果，不得分别手工填写。

### 7.4 质量门槛

综合得分用于排序，但不能覆盖关键风险。出现以下情况时，该配置必须标记为“不建议作为当前完整系统配置”，并在报告中说明具体样例：

- 存在泄露个人敏感信息或声称能够查询未授权个人数据的严重输出。
- 对赔付、时效、禁限寄等事项作出明确虚假承诺。
- 正式运行中出现大面积空输出、不可解析输出或持续请求失败。
- RAG 回答的关键结论与检索依据明显冲突，且可能误导业务处理。

## 8. 执行流程

1. 固定题集版本、评分配置、运行环境和待测模型清单。
2. 对模型及四种系统组合执行真实评估，保存未经改写的原始输出和检索记录。
3. 计算确定性自动指标，并对异常、空输出和解析失败进行结构化标记。
4. 按统一 rubric 完成人工评分与抽样复核，保留评审记录。
5. 从明细结果生成模型级、组合级、维度级和题型级汇总数据。
6. 使用汇总数据生成统一雷达图、对比图和错误案例表。
7. 自动填充报告中的数据表和图表，人工只负责结论复核与文字分析。
8. 执行一致性检查，确认报告分数、汇总 JSON、CSV 和图表完全一致。

若某次模型调用、检索或评分未完成，应明确标记为失败或缺失并说明原因。不得删除失败样例后重新计算一个看似完整的分数。

## 9. 工程设计要求

### 9.1 单一数据链路

评估工程应保持一条正式数据链路：题集加载、真实执行、单题评分、指标聚合、图表生成和报告生成依次传递结构化数据。自动评估、雷达图和报告不得各自实现重复的指标计算逻辑。

### 9.2 模块职责

- Runner 只负责调用模型和完整系统，并记录原始事件。
- Scorer 只负责根据题型和 rubric 计算单题指标。
- Aggregator 只负责聚合维度分数、综合分数和统计量。
- Renderer 只读取聚合结果生成图表与报告，不重新判分。
- Provider 与向量库继续复用项目已有抽象，不在评估代码中写死具体服务。

公共协议使用清晰的数据模型约束，新增模型、指标或图表时通过扩展实现完成，避免修改无关模块。相同的 JSON 提取、文本归一化、风险检测、RAG 路由和指标换算逻辑只能保留一个权威实现。

### 9.3 注释与可维护性

新增 Python 代码文件开头必须使用中文说明文件意图、输入、输出和在评估流程中的职责。函数必须提供中文 docstring，说明参数、返回值、副作用和异常；复杂评分公式、边界处理、数据兼容逻辑及关键业务判断必须增加中文注释。

注释用于解释设计原因和不明显的约束，不重复描述一眼可见的语句。实现应遵循 SOLID 与 DRY，禁止为了临时运行复制评分逻辑或建立第二套报告数据源。

## 10. 雷达图要求

### 10.1 主图与输出路径

- 主雷达图固定输出到仓库统一图片目录：`images/model_overall_evaluation_radar.jpg`。
- `week7/README.md` 直接使用 `../images/model_overall_evaluation_radar.jpg` 引用主图，不复制图片，不建立 symlink。
- 主图展示当前完整系统配置，即当前推荐 LoRA 模型与真实 RAG 链路组合的七维得分。
- 主要模型或系统组合另行生成对比雷达图；候选配置较多时拆分为 Base/LoRA 对比和 RAG 开关对比，避免曲线相互遮挡。
- 图表文件名必须稳定。重复生成同一周期、同一配置的图表时覆盖对应产物，禁止通过复制文件产生多个内容相同的版本。

### 10.2 坐标与数据规则

- 雷达图固定展示七个一级维度，顺序为：邮政业务正确性、RAG 检索与依据质量、任务完成与可用性、指令与格式遵循、多轮对话一致性、安全与可信边界、运行效率与稳定性。
- 所有维度量程统一为 0 至 100，径向刻度固定为 20、40、60、80、100，不根据当次得分动态缩放。
- 第一维从正上方开始，各维度按顺时针排列；首尾数据点必须闭合，不能遗漏最后一个维度。
- 雷达图数据只能读取版本化聚合结果文件；不得在绘图代码、Markdown 或图片中手写分数。
- 图中得分与 `metrics.json`、`metrics.csv` 及最终报告必须来自同一聚合结果，并通过一致性检查。
- 缺少有效数据的维度不得按 0 分或 100 分填充，应中止正式图表生成并报告缺失字段。

### 10.3 视觉与排版

- 图表标题、维度名称、图例和注释统一使用中文，并显式配置可用中文字体，避免生成方框或乱码。
- 主图仅展示一个完整系统系列，使用清晰实线、可辨识节点和低透明度填充；对比图每张最多展示四个系列。
- 不同系列同时使用颜色、线型和节点形状区分，保证彩色屏幕、灰度打印和常见色觉差异下均可辨认。
- 维度标签不得与图形、标题、图例或图片边缘重叠；长标签允许合理换行，但不能缩写成无法理解的内部名称。
- 图例必须写明模型规模、Base/LoRA 和 RAG 状态，不能只写无法追溯的简称。
- 画布应为接近正方形的稳定尺寸，并预留中文长标签空间；保存前执行布局调整，确保 README、文档站和 PDF 中均可完整显示。
- 输出格式使用 JPG，与仓库现有报告图片格式保持一致；背景使用白色，保证文档站和打印报告中的可读性。
- 图片本身只承载图表信息，不在图中堆叠大段结论；详细解释放在 Markdown 报告正文。

### 10.4 图表验收

- 七个维度、刻度、标题、图例和配置名称完整且无错别字。
- 主图能够直接回答当前完整系统各维度表现，不混入诊断运行或 mock 数据。
- 对比图中的每个系列都能回查到唯一运行 ID、模型配置和聚合结果。
- 图片生成后检查非空、无裁切、无文字重叠、无中文乱码，并确认 README 引用路径有效。
- 修改评分或聚合结果后重新生成图表，禁止只替换报告数字而保留旧图。

## 11. 最终报告结构

每次正式测评生成一份中文报告，至少包含：

1. 测评摘要：当前推荐配置、综合得分、主要优势和主要限制。
2. 测评范围：模型、adapter、RAG/SFT 组合、题集和运行环境。
3. 评分方法：七个维度、权重、指标和质量门槛。
4. 综合结果：总分排名、七维雷达图和配置对比表。
5. 分维度分析：业务、RAG、任务、格式、多轮、安全和运行指标。
6. RAG 专项分析：路由分布、Recall@K、MRR、引用有效率和典型案例。
7. 典型样例：表现良好的案例、可改进案例和风险案例，保留题目 ID。
8. 与上次结果对比：得分变化、配置变化和题集版本变化。
9. 结论与改进建议：按影响和实施成本给出后续方向。
10. 附录：完整配置、数据版本、计算公式、失败记录和产物索引。

报告语言应清楚、客观，优先解释结果对系统使用的实际意义。不得只堆叠指标，也不得将有限题集上的结论描述为绝对能力。

## 13. 验收标准

- 已真实运行并记录计划内所有可用模型和系统组合；缺失项有明确原因。
- 七个雷达图维度均有定义、样例、指标、权重和可回查的原始记录。
- RAG 得分覆盖 Router、召回、排序、引用和回答依据，不以生成答案分数替代检索评估。
- 自动评分与人工评分边界清楚，开放式回答没有被不适用的字符串匹配主导。
- 同一份聚合结果能够生成 JSON、CSV、雷达图和 Markdown 报告，数值完全一致。
- 报告包含配置对比、典型案例、限制说明和可执行的改进建议。
- 正式报告中不存在 mock、随机或人工手填的模型分数。
- 代码符合 SOLID 与 DRY，复用现有 Provider、RAG 和评估能力，没有平行重复实现。
- 新增代码具备完整中文文件意图注释、函数注释和关键逻辑注释。
- 评分与聚合测试、失败路径测试、报告一致性检查全部通过。
- Markdown 报告可以通过统一报告构建入口渲染，雷达图和中文内容显示正确。
