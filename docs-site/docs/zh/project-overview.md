# 项目总览

这个项目围绕邮政客服场景展开，但真正的主线不是页面本身，而是两件事：

1. 把邮政知识整理成可检索、可引用的 RAG 链路。
2. 把模型微调到更贴近邮政客服场景的状态。

Django 页面、API、推理部署和负载均衡这些内容都存在，但它们更多是承接和演示这两条主线，而不是项目的核心目标。

整体可以按三段来理解：

## 第一阶段：数据筛选与知识整理

这一阶段主要处理邮政客服数据的筛选、分析和知识整理，包括：

1. 从泛客服数据中筛出邮政相关对话。
2. 做关键词统计、样本分析和边界 case 检查。
3. 为后续知识库和训练数据准备更干净的输入。

对应目录主要是：

- `week1/`
- `week1-module-Web-Crawler/`

## 第二阶段：RAG 智能客服系统

这一阶段的重点是把知识整理和问答系统接起来，形成可以运行、可以引用知识来源的邮政智能客服链路。

这一阶段保留的重点包括：

1. RAG 检索链路整理。
2. Django 应用接入。
3. 会话、引用、工单等交互能力组织。
4. 不同模式下的效果对比入口。

对应目录主要是：

- `week2/`

## 第三阶段：LoRA 微调与推理接入

这一阶段主要是围绕 LoRA 微调和推理接入展开，对 Qwen2.5 做邮政客服场景适配，并保留一版统一可复述的实验结论。

对应目录主要是：

- `week3/`

当前仓库里保留的是整理后的实验工程、报告和说明文档，原始训练环境中的大量运行产物没有继续纳入版本库。

如果只看当前整理版，项目主次关系可以直接理解为：

1. `RAG` 是知识能力主线。
2. `LoRA` 是模型能力主线。
3. Django 页面与部署链路主要用于把这两部分能力接成一个可验证的系统 demo。
