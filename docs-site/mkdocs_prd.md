# MkDocs 双语文档站点建设说明

## 1. 文档说明

这部分文档站点工作当时已经在公司电脑上完成过一版，当前仓库里的 `docs-site` 目录，是在原始环境和部分文档数据无法完整迁出的情况下，按已经做完的结构重新整理出来的版本。

所以这里记录的不是一份“待建设 PRD”，而是已经采用过的站点结构、组织方式和建设口径。

## 2. 站点建设目标

这套文档站点当时主要服务两个目标：

1. 把项目资料整理成结构清楚、可继续扩展的双语文档工程。
2. 把文档构建和部署从主项目目录中独立出来，便于长期维护。

站点本身采用 `MkDocs`，多语言部分按 `i18n` 方案组织，核心原因还是维护成本更低，结构也更稳定。

## 3. 为什么会有当前这版重建目录

当前仓库里的 `docs-site/` 不是把公司电脑上的完整站点直接拷贝过来的版本，而是基于当时已经做完的结构重新复现出来的整理版。

原因很实际：原始环境、部分配置上下文和文档数据没有办法完整迁出，所以现在这套内容更接近“按完成结果重建”，而不是重新从零设计一套新方案。

## 4. 站点组织方式

文档站点整体按独立工程组织，相关配置、内容和构建逻辑尽量收敛在 `docs-site/` 内，不再分散在仓库多个无关位置。

当时采用这套组织方式，主要是为了把边界收清楚：

1. 文档站点是一套单独可维护的工程。
2. 文档内容和主项目业务代码解耦。
3. 后续迁移、扩展或补内容时，不需要反复改主项目结构。

## 5. 双语结构

双语部分按中文和英文平行组织，中英文导航尽量保持一致。这样做不是为了形式统一，而是为了后续维护时不出现一边有、一边缺的结构漂移。

当时的处理原则比较明确：

1. 每种语言都有独立入口。
2. 相同主题的页面尽量一一对应。
3. 导航层级尽量保持一致。
4. 允许早期使用占位页，但骨架先搭完整。

## 6. 信息架构

站点的信息架构当时按稳定栏目来搭，而不是按临时文件堆出来。大致会覆盖这几类内容：

1. 首页与总览。
2. 快速开始或入门入口。
3. 用户指南或专题说明。
4. 项目说明与协作约束。
5. 过程文档、需求文档和补充记录。

这样做的好处是，后续补内容时直接往既有栏目里填，不需要频繁推翻导航结构。

## 7. 与当前项目主线的对应关系

这套文档站点服务的仍然是整个邮政实习项目，而不是孤立存在的一块文档工程。

当前项目主线可以按阶段理解：

1. `week1`：邮政客服数据筛选、样本分析、规则整理和知识沉淀。
2. `week2`：RAG 智能客服系统。
3. `week3`：本地 Apple MLX 路线的 LoRA 微调实验。

当前整理口径下，LoRA 结果不再展开成大量 rank sweep 过程结论，而是直接保留一个统一、便于说明的版本：

1. `Qwen2.5-3B` 以 `rank 2` 作为当前结论。
2. `Qwen2.5-7B` 以 `rank 4` 作为当前结论。

这里默认承认实验本身存在一定随机性，因此当前文档只保留这一版整理后的结论，不再把所有波动都写进文档站点首页逻辑里。

## 8. 配置与构建口径

MkDocs 这部分当时采用的是比较标准的配置方式，重点不是做复杂定制，而是保证以下几件事：

1. 站点信息清楚。
2. 导航结构稳定。
3. 多语言切换明确。
4. 构建依赖集中。
5. 后续自动部署容易接入。

这也是为什么当前说明里仍然保留 `MkDocs + i18n + GitHub Actions` 这条组合：这套结构在当时已经验证过，继续沿用成本最低。

## 9. 当前目录的意义

当前仓库里的 `docs-site/` 更适合理解为：

1. 对原始文档工程的一次重建整理。
2. 对当时完成结果的一次结构化保留。
3. 后续如果继续补文档站点时的统一入口。

因此它不是临时草稿目录，也不是纯粹的占位目录，而是按既有完成结构复原出来的文档工程起点。

## 10. 总结

这套 MkDocs 双语文档站点方案在原始项目推进中已经做完过一版，当前仓库中的内容是基于历史完成结果重建出来的整理版本。

保留这份说明，主要是为了把几件事说清楚：

1. 文档站点这件事已经做过。
2. 当前版本是重建整理版，不是从零构思的新 PRD。
3. 站点结构、双语组织方式和项目主线之间的对应关系已经明确。
4. 后续如果继续恢复配置或补内容，可以直接沿着当前目录往下做，不需要重新定义一遍基础结构。
