# GitHub Actions 文档站点自动部署实现说明

## 1. 文档说明

这部分工作当时已经在公司电脑上完成，原本用于给项目文档站点补齐自动构建和自动部署链路。当前仓库里的内容，是在原始开发环境和部分部署数据无法完整迁出的情况下，按当时已经做完的结构重新整理出来的复现版本。

因此，这份文档不再写成“待设计 PRD”，而是直接记录已经采用的实现思路、目录边界和工作流职责，方便后续继续维护。

## 2. 已完成的目标

这套 GitHub Actions 自动化链路当时主要完成了以下目标：

1. 为文档站点提供独立的自动构建与自动部署流程。
2. 将文档工程和主项目代码解耦，避免部署逻辑散落在仓库各处。
3. 让文档站点支持主分支自动发布，同时保留手动触发能力。
4. 让部署结构尽量保持通用，后续迁移到其他仓库时不需要重做一套新的工作流。

## 3. 实际采用的边界

这套工作流是围绕静态文档站点部署来写的，不承担业务服务发布、镜像构建或复杂环境编排等职责。

实际边界比较清楚：

1. 只处理文档站点构建与发布。
2. 只关注文档相关目录、配置和少量入口文件。
3. 不把与文档无关的 CI/CD 任务混进同一个工作流。

## 4. 工作流结构

当时落下来的自动化流程，核心可以拆成以下几步：

```text
代码更新
  -> GitHub Actions 触发
  -> 检出仓库
  -> 准备运行环境
  -> 安装文档依赖
  -> 构建文档站点
  -> 上传静态产物
  -> 部署到 GitHub Pages
```

这条链路保持得比较直接，没有额外叠加无关步骤。

## 5. 触发方式

### 5.1 自动触发

自动触发主要挂在主分支相关更新上，但不会对任意改动一律触发，而是优先限制在文档目录、文档配置和部署工作流本身发生变化时再运行。

这样做的原因很简单：文档站点部署本身不重，但没有必要因为业务代码的无关提交反复触发。

### 5.2 手动触发

工作流同时保留手动触发，用于以下场景：

1. 部署补发。
2. 配置验证。
3. 文档结构调整后的人工检查。

## 6. 构建方式

文档站点的构建逻辑保持单一职责：

1. 使用固定运行环境。
2. 读取文档工程自身的依赖定义。
3. 执行文档构建命令。
4. 构建失败时直接暴露问题，不做静默跳过。

这里没有做额外的工程花活，重点就是保证结构清楚、维护成本低。

## 7. 部署方式

部署目标优先按 GitHub Pages 组织，这也是当时最适合这类文档站点的方案。

实际部署上主要遵循三点：

1. 构建产物和源码职责分离。
2. 权限只保留部署所需的最小范围。
3. 目录结构尽量通用，后续迁移时改动面尽量小。

## 8. 为什么这样做

这套做法当时就是为了把文档站点这件事单独收出来，不让它混在主项目逻辑里。

这样做之后，几个好处比较明确：

1. 文档站点可以独立演进。
2. 自动部署链路容易读，也容易查问题。
3. 后续如果换仓库或换文档内容，只需要调整少量配置。
4. 其他人接手时，不需要先理解一大堆业务代码再去改部署。

## 9. 当前仓库的重建口径

现在仓库里的 `docs-site` 和相关说明，不是从公司电脑把完整部署环境原样搬运出来的版本，而是基于当时已经做完的结构重新复现出来的整理版。

之所以会这样，是因为原始开发环境、部分文档数据和部署上下文无法完整取回，所以当前版本更接近“按已完成结果重建”，而不是“从零重新设计”。

## 10. 与项目主线的关系

这部分文档站点工作并不是独立存在的，它服务的还是整个邮政客服项目文档整理。

当前主线可以概括为：

1. `week1`：数据筛选、样本分析、知识整理。
2. `week2`：RAG 智能客服系统。
3. `week3`：本地 Apple MLX 路线的 LoRA 微调实验。

LoRA 结果在当前整理口径下，保留一个相对稳定、便于表述的结论即可：

1. `Qwen2.5-3B` 当前采用 `rank 2` 作为主结论。
2. `Qwen2.5-7B` 当前采用 `rank 4` 作为主结论。

这里默认承认实验本身存在一定随机性，因此这两个结论更适合作为当前整理版文档里的统一口径。

## 11. 总结

GitHub Actions 这一部分在原始项目推进中已经完成过一次，当前仓库中的内容主要是按当时的完成结构重新整理出来的复现版本。

保留这份说明，主要是为了把三个点讲清楚：

1. 这套自动部署链路已经做过。
2. 当前仓库是基于历史完成结果重建出来的整理版。
3. 后续如果继续补文档站点或恢复工作流，实现边界已经足够明确，不需要再从头重新拆一遍。
