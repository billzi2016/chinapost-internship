# UI Plan

## 1. 页面目标

做一个类似 OpenAI GPT 的正式聊天窗口：

- 左侧是历史会话和模式控制。
- 右侧是当前聊天窗口。
- AI 回复支持 Markdown。
- 每次回复展示引用了哪些对话。
- 等待 AI 时有明确加载动画。
- 最终生成并展示工单 JSON。

## 1.1 前端实现边界

本项目不是前后端分离项目。

实现方式：

- 使用 Django templates 做页面骨架。
- 可以参考 Chatbot UI / Open WebUI 的布局方式。
- 使用 Bootstrap 或少量自写 CSS 实现左侧历史 + 右侧聊天。
- 使用原生 JavaScript 处理 SSE、发送消息、状态更新。
- Markdown 使用成熟 JS 库渲染。
- 不引入 React、Vue、Next.js。
- 不引入独立前端构建链。
- 不把完整 ChatGPT clone 项目搬进来。

原则：

- 复用成熟 UI 组件和库。
- 不造前端框架。
- 不为了界面引入超出 Django template 项目规模的架构。
- 不手写容易出问题的基础交互，例如拖拽改宽、复杂滚动同步、Markdown 安全清洗、代码高亮。
- 对这类能力优先使用成熟小库或 Bootstrap 现成组件，并固定版本。

## 2. 使用者

使用者是邮政客服或实习项目审核人员。

页面要像一个内部业务工具：

- 清晰。
- 稳定。
- 正式。
- 便于连续操作。
- 不做营销页，不做花哨 hero。

## 3. 布局

```text
┌──────────────────────────┬──────────────────────────────────────────────┐
│ 历史会话                  │ 当前聊天                                      │
│                          │                                              │
│ + 新建会话                │ 消息列表                                      │
│                          │                                              │
│ 会话 A                    │ 用户消息                                      │
│ 会话 B                    │ AI 回复 markdown                             │
│ 会话 C                    │ 引用对话                                      │
│                          │ 工单 JSON                                     │
│ ──────────────────────── │                                              │
│ ☑ 检索增强生成（RAG）     │                                              │
│ ☐ 监督微调模型（SFT）     │ 输入框 + 发送按钮                              │
│   当前不存在 SFT 模型     │                                              │
└──────────────────────────┴──────────────────────────────────────────────┘
```

左侧历史栏宽度：

- 可以支持拖拽调整宽度。
- 拖拽能力不要手写。
- 使用成熟的 split-pane / resizable pane 小库。
- 必须固定版本。
- 必须能在纯 Django template + 原生 JS 页面中使用。
- 如果现成库引入成本过高，则先使用固定宽度，不为了拖拽写脆弱自定义逻辑。

## 4. 左侧区域

### 4.1 历史会话

展示：

- 新建会话按钮。
- 会话列表。
- 当前选中会话。
- 会话标题由 `gpt-oss:20b` 总结生成。

标题规则：

- 中文。
- 简短。
- 不超过 20 个汉字。
- 示例：“包裹派送异常咨询”。

### 4.2 模式勾选

左侧底部放两个勾选项：

- `检索增强生成（RAG）`
- `监督微调模型（SFT）`

行为：

- 勾选 RAG：本轮聊天使用 pgvector 向量检索。
- 取消 RAG：本轮聊天只使用当前会话上下文。
- 勾选 SFT：切换到监督微调模型模式。
- 当前没有 SFT 模型，所以勾选后在下方显示红色提示：

```text
当前不存在 SFT 模型
```

后端也要校验 SFT 状态，不能只靠前端提示。

关键规则：

- `use_sft=true` 时，后端必须按 SFT provider 和 SFT 模型处理。
- 如果 SFT 模型不存在，本轮发送后直接显示错误，不生成普通回答。
- 不允许自动回退到 `gpt-oss:20b`。
- 用户取消 SFT 勾选后，再恢复普通 `gpt-oss:20b` 模式。

## 5. 右侧聊天区域

### 5.1 消息展示

消息类型：

- 用户消息
- AI 消息
- 系统错误提示
- 引用对话
- 工单 JSON

AI 消息：

- 支持 Markdown。
- 保留段落、列表、代码块、表格。
- 使用成熟 Markdown 渲染库。
- 渲染前必须做 sanitizer。

### 5.2 引用对话

每次 RAG 回复都显示引用。

展示字段：

- 引用编号。
- 相似度。
- 来源 ID 或来源路径。
- 对话片段。

展示形式：

- 放在 AI 回复下方。
- 默认显示摘要。
- 可以展开查看完整片段。

### 5.3 工单 JSON

AI 回复完成后展示工单 JSON。

展示方式：

- 放在回复下方独立区域。
- 使用等宽字体。
- 支持复制。
- 如果 JSON 不合法，显示错误原因。

字段：

```json
{
  "user_id": "",
  "timestamp": "",
  "service_type": "",
  "issue_type": "",
  "user_request": "",
  "summary": "",
  "resolution": "",
  "need_follow_up": false
}
```

## 6. 输入区

位置：

- 固定在右侧底部。

包含：

- 多行输入框。
- 发送按钮。
- Enter 发送。
- Shift + Enter 换行。

发送时：

- 禁用发送按钮。
- 显示三点跳跃动画。
- SSE 收到 `done` 后恢复。

## 7. 等待动画

等待 AI 时显示三个点跳跃。

视觉要求：

- 低干扰。
- 位于 AI 消息气泡内。
- 不改变布局高度太多。

示意：

```text
● ● ●
```

动画：

- 三个点依次上下跳动。
- 支持 `prefers-reduced-motion`，减少动画。

## 8. 视觉风格

定位：邮政内部业务系统。

设计方向：

- 固定亮色主题。
- 不使用黑暗背景。
- 不默认提供暗色主题。
- 低饱和灰白底。
- 使用邮政绿色作为主强调色。
- 使用红色只表达错误或不可用状态，例如 SFT 不存在。
- 控件紧凑，适合长时间工作。

建议色板：

- 页面背景：`#F6F7F4`
- 面板背景：`#FFFFFF`
- 主文字：`#1E2723`
- 次级文字：`#66736D`
- 邮政绿色：`#0B7A3B`
- 分割线：`#DDE3DE`
- 错误红：`#B42318`

字体：

- 中文正文：系统默认 sans-serif。
- JSON / 代码：`SFMono-Regular`, `Consolas`, monospace。

控件：

- 卡片圆角不超过 8px。
- 按钮、输入框、引用区尺寸稳定。
- 文本不能溢出按钮或侧栏。

## 9. CSS 和 JS 版本

所有静态资源引用都带版本：

```django
{% load static %}
<link rel="stylesheet" href="{% static 'web/css/chat.css' %}?v={{ static_version }}">
<script src="{% static 'web/js/chat.js' %}?v={{ static_version }}" defer></script>
```

`static_version` 来自 Django context processor 或 settings。

Bootstrap 和 Markdown 渲染库也必须通过明确版本引入。

建议：

- 优先使用本地 vendored 静态文件。
- 如果使用 CDN，必须固定版本号。
- 不使用浮动版本，例如 `latest`。
- 可选引入成熟 resizable pane 小库处理左侧栏拖拽宽度。
- 可选引入成熟代码高亮库处理 Markdown code block。

## 10. SSE 前端流程

发送消息后：

1. 前端创建用户消息。
2. 前端创建 AI 占位消息。
3. 显示三个点动画。
4. 打开 SSE 连接。
5. 收到 `citation`，更新引用区域。
6. 收到 `delta`，追加 AI Markdown 原文。
7. 持续重新渲染 Markdown。
8. 收到 `ticket`，展示工单 JSON。
9. 收到 `done`，关闭等待动画。
10. 失败时显示错误消息。

## 11. 空状态

无会话时：

- 右侧显示空聊天窗口。
- 输入框可直接输入。
- 发送后自动创建新会话。

不要写大段说明文案，界面以可操作为主。

## 12. 验收标准

- 页面打开后第一屏就是聊天工具。
- 左侧历史会话和模式勾选清楚可见。
- RAG 勾选能影响请求参数。
- SFT 勾选后红色提示立即显示。
- AI 等待时有三点跳跃动画。
- SSE 回复能逐字/逐段显示。
- Markdown 正确渲染。
- 引用对话能看到来源和内容。
- 工单 JSON 能被前端解析并格式化展示。
