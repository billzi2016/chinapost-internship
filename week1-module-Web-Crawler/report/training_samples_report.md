# 邮政 FAQ 训练样本构建报告

## 1. 报告目标

本报告分析 `week1-module-Web-Crawler/data/parsed/training_samples.jsonl` 中已经抽取完成的邮政 FAQ 训练样本。该文件由网页爬虫和解析流程生成，目标是把中国邮政 EMS 公开 FAQ 页面中的问答内容整理成结构化样本，作为后续邮政客服模型 SFT、评估集构造和知识库入库的基础数据。

这份报告不只记录样本条数，而是完整说明：

1. 数据从哪里来。
2. 每条 JSONL 样本包含哪些字段。
3. 当前 11 条样本分别覆盖什么业务场景。
4. 这些样本能不能直接用于 SFT。
5. 当前数据的不足和后续扩展方向。

## 2. 数据文件概况

输入文件：

```text
week1-module-Web-Crawler/data/parsed/training_samples.jsonl
```

文件格式为 JSON Lines，一行是一条训练样本。当前共包含 11 条样本，全部来自中国邮政 EMS FAQ 网络页面。

| 项目 | 内容 |
|---|---|
| 样本数量 | 11 |
| 数据来源 | 中国邮政 EMS FAQ |
| 来源标识 | `ems_policy_network` |
| 公司字段 | 中国邮政 EMS |
| 文件格式 | JSONL |
| 主要用途 | 邮政客服知识抽取、SFT 样本构造、评测题构造、RAG 知识库入库 |

## 3. 字段结构

每条样本包含 14 个字段：

| 字段 | 类型 | 含义 |
|---|---|---|
| `source_id` | string | 数据源标识，当前为 `ems_policy_network` |
| `company` | string | 所属机构，当前均为中国邮政 EMS |
| `url` | string | 原始 FAQ 页面链接或锚点链接 |
| `title` | string | FAQ 标题，可作为用户问题或知识点标题 |
| `published_at` | string | 发布时间，当前样本中为空 |
| `policy_categories` | list[string] | 业务类别标签，例如国际清关、保价赔付、时限标准 |
| `summary` | string | 解析后的问答摘要，适合转成 SFT assistant 答案 |
| `evidence_text` | string | 原始证据文本，用于追溯和核验 |
| `insurance_available` | bool | 是否涉及可保价或保险服务 |
| `insurance_type` | string | 保险或保价类型，未知时为 `未知` |
| `compensation_limit` | string | 赔付上限，当前样本未抽到明确数值 |
| `claim_deadline` | string | 申诉或理赔期限，当前样本未抽到明确期限 |
| `requirements` | list[string] | 办理要求或材料要求，当前样本为空数组 |
| `insurance_exclusions` | list[string] | 保险或赔付排除项，当前样本为空数组 |

字段设计上保留了“原文证据”和“结构化业务字段”两类信息。`summary` 适合直接进入训练样本，`evidence_text` 用于事后检查模型答案是否有出处，保价和理赔相关字段用于后续构造赔付、安全边界和合规问答。

## 4. 类别覆盖情况

当前 11 条样本覆盖 5 类业务标签。由于一条样本可以有多个标签，因此标签计数总和大于样本数。

| 类别 | 样本数 | 说明 |
|---|---:|---|
| 国际清关 | 6 | 覆盖超值超量、滞留海关、网上申报、缴税放行、进口关税、水陆路包裹时限 |
| 保价赔付 | 3 | 覆盖投诉/理赔进度查询，分别对应微信公众号、支付宝生活号、邮政 EMS APP |
| 产品服务 | 2 | 覆盖即日专递、国内标准快递 |
| 时限标准 | 2 | 覆盖即日专递和水陆路包裹时限 |
| 资费规则 | 1 | 覆盖国际/港澳台进口邮件关税咨询 |

从覆盖面看，当前样本更偏向国际清关和售后查询，已经能支持少量邮政客服垂直知识训练；但样本总量仍然很小，还不能支撑完整客服模型训练。

## 5. 全量样本清单

### 样本 1：即日专递产品介绍

| 字段 | 内容 |
|---|---|
| 来源 | 中国邮政 EMS |
| 类别 | 产品服务、时限标准 |
| 是否涉及保险 | 否 |
| 标题 | 即日专递产品介绍 |

摘要：

> 即日专递业务面向所有用户开办，主要依托自主运输网，在特定线路提供高效、安全的当日上午寄件、当日送达的寄递服务，以同城流向为主。具体开办范围请进入人工客服进行查询。

分析：这条样本适合构造“即日专递是什么”“是否当日送达”“哪些线路能寄”等问答。它的边界也比较明确：具体开办范围需要转人工或查询官方渠道，模型不能自行编造覆盖城市。

### 样本 2：国内标准快递产品介绍

| 字段 | 内容 |
|---|---|
| 来源 | 中国邮政 EMS |
| 类别 | 产品服务 |
| 是否涉及保险 | 否 |
| 标题 | 国内标准快递产品介绍 |

摘要：

> 国内标准快递业务面向所有客户开办，主要依托陆路运输，提供高效、安全的省际间（不含临近省份）物品寄递服务。具体开办范围请进入人工客服进行查询。

分析：这条样本可以训练模型解释国内标准快递的服务定位。需要注意“不含临近省份”这个限制，回答时不能泛化为所有省际或所有国内线路。

### 样本 3：微信公众号查询投诉/理赔进度

| 字段 | 内容 |
|---|---|
| 来源 | 中国邮政 EMS |
| 类别 | 保价赔付 |
| 是否涉及保险 | 否 |
| 标题 | 如何查询投诉/理赔进度 |

摘要：

> 可通过“EMS中国邮政速递物流微信公众号-我的-个人中心-自助投诉理赔-我的售后”实时查看处理过程与结果。

分析：这条样本适合训练售后进度查询路径。它只说明查询入口，不说明赔付金额或赔付承诺，因此模型回答时应避免承诺“必赔”“多久到账”。

### 样本 4：海关定义邮件内件超值超量怎么办

| 字段 | 内容 |
|---|---|
| 来源 | 中国邮政 EMS |
| 类别 | 国际清关 |
| 是否涉及保险 | 否 |
| 标题 | 海关定义邮件内件超值超量怎么办 |

摘要：

> 对于内件数量或价值超过海关限定的邮件，会被海关定为超值超量邮件，此类邮件需要收件人办理退运或者按货物办理通关手续。

分析：这条样本明确给出两种处理方向：退运，或按货物办理通关。模型需要把决定权归于海关和收件人办理流程，不能擅自判断一定可以正常派送。

### 样本 5：邮件滞留海关如何处理

| 字段 | 内容 |
|---|---|
| 来源 | 中国邮政 EMS |
| 类别 | 国际清关 |
| 是否涉及保险 | 否 |
| 标题 | 邮件滞留海关如何处理 |

摘要：

> 海关部门对于无法预判价值或价值较高的邮件都会进行查验，一般最长不超过一个月。

分析：这条样本可以训练模型解释海关查验导致的滞留。回答时要强调海关查验和价值判断，而不是把滞留直接归因于快递运输异常。

### 样本 6：国际/港澳台地区邮件网上能办理申报吗

| 字段 | 内容 |
|---|---|
| 来源 | 中国邮政 EMS |
| 类别 | 国际清关 |
| 是否涉及保险 | 否 |
| 标题 | 国际/港澳台地区邮件网上能办理申报吗 |

摘要：

> 目前国内大部分关区已经支持网上申报、缴税、退运。因各关区网上申报有所差异，具体申报方式以查验通知短信或报关通知单上内容为准，或联系当地报关行确认。

分析：这条样本的关键是“多数支持，但各关区有差异”。模型需要给出可操作路径：查看短信或报关通知单，必要时联系当地报关行。

## 6. 可转化的 SFT 形式

这些样本不是最终 SFT JSONL，而是“训练原料”。后续可以按以下方式转成对话样本。

### 6.1 单轮问答

适合产品介绍、时限解释、清关流程说明。

```json
{"messages":[{"role":"system","content":"你是中国邮政 EMS 客服助手，回答必须基于官方 FAQ，不确定时引导用户查询官方渠道。"},{"role":"user","content":"即日专递是什么？能当天送到吗？"},{"role":"assistant","content":"即日专递业务面向所有用户开办，主要依托自主运输网，在特定线路提供当日上午寄件、当日送达的寄递服务，以同城流向为主。具体开办范围建议进入人工客服或官方渠道查询。"}]}
```

### 6.2 多入口整合问答

适合投诉/理赔进度查询这种同标题、多渠道样本。

```json
{"messages":[{"role":"system","content":"你是中国邮政 EMS 客服助手，回答售后问题时优先给出官方入口。"},{"role":"user","content":"我想查投诉或者理赔进度，在哪里看？"},{"role":"assistant","content":"可以通过以下官方入口查询处理过程与结果：1. EMS中国邮政速递物流微信公众号-我的-个人中心-自助投诉理赔-我的售后；2. EMS中国邮政速递物流支付宝生活号-我的-个人中心-自助投诉理赔-我的售后；3. 邮政EMS APP-投诉理赔-我的售后。"}]}
```

### 6.3 边界控制问答

适合关税、清关、赔付等不能由模型自行承诺的场景。

```json
{"messages":[{"role":"system","content":"你是中国邮政 EMS 客服助手，涉及海关、税费和赔付金额时不得编造结论。"},{"role":"user","content":"我的进口包裹要交多少关税？你直接帮我算一下。"},{"role":"assistant","content":"关税由海关部门收取，EMS 只是代收，具体收费标准需要以海关规定和实际审核结果为准。你可以到海关网站查询，或拨打海关客服电话 12360 咨询。"}]}
```

## 7. 数据质量检查

### 7.1 优点

1. 每条样本都保留了 `url` 和 `evidence_text`，方便追溯原始 FAQ。
2. `policy_categories` 已经把产品、清关、赔付、时限、资费等场景分开，后续可以按类别采样。
3. 同一问题的不同官方渠道被保留下来，例如投诉/理赔进度查询的微信公众号、支付宝生活号和 APP。
4. 对不能直接确定的事项，样本原文已经包含“进入人工客服”“以通知短信或报关通知单为准”“联系报关行”“咨询海关 12360”等边界提示。

### 7.2 不足

1. 样本量只有 11 条，不能单独支撑模型微调，只能作为垂直领域种子样本或评估样本。
2. `published_at` 为空，无法判断 FAQ 的发布时间和时效性。
3. `insurance_available` 均为 `false`，但其中 3 条属于投诉/理赔进度查询，这说明当前字段只能表达“是否有保险/保价产品信息”，不能直接表达“是否属于售后理赔话题”。
4. `compensation_limit`、`claim_deadline`、`requirements`、`insurance_exclusions` 当前均为空，赔付金额、申诉期限、材料要求、免责条款还需要继续补数据。
5. 当前样本的语言风格混合了 FAQ 原文和客服话术，例如部分样本带有“亲”，后续进入 SFT 前需要统一语气。

## 8. 对后续微调的价值

这批样本最适合承担三类作用。

第一，作为邮政客服知识种子。它覆盖了清关、时效、投诉理赔入口、产品服务等常见咨询类型，可以让模型学习到邮政客服中高频且高风险的表达边界。

第二，作为评测集来源。因为每条样本都有明确答案，可以转成选择题、JSON 格式题或短问答题，用来检查模型是否能保持官方口径。例如“关税由谁收取”“超值超量邮件如何处理”“投诉/理赔进度在哪里查”都适合作为自动评测题。

第三，作为 RAG 知识库入库单元。`title` 可以作为检索标题，`summary` 可以作为回答片段，`evidence_text` 和 `url` 可以作为引用来源。对于清关和资费类问题，RAG 比直接 SFT 更安全，因为政策内容可能变化。

## 9. 后续扩展建议

1. 扩大爬取范围，补充更多中国邮政、EMS、国际件、保价、赔付、禁限寄、时限、资费和网点服务 FAQ。
2. 对重复问题做聚合，例如把 3 条“如何查询投诉/理赔进度”合并为一条多渠道标准答案。
3. 增加字段校验脚本，检查 `summary` 是否为空、`evidence_text` 是否可追溯、`policy_categories` 是否在枚举范围内。
4. 把当前 JSONL 转成三种下游格式：SFT 对话样本、自动评估选择题、RAG 文档片段。
5. 对政策变化风险高的内容标注“需实时查询”，例如关税、清关要求、开办范围和赔付规则。
6. 对涉及赔付和税费的样本增加安全模板，强制模型输出“以官方审核结果为准”，避免生成未经证实的承诺。

## 10. 结论

`training_samples.jsonl` 当前规模不大，但字段结构完整，已经具备从网页 FAQ 到训练数据的基本闭环。它的主要价值不是直接大规模训练，而是证明爬虫、解析、结构化、追溯和下游 SFT/评估/RAG 转换链路已经打通。

后续只要继续扩大数据量，并补充赔付金额、禁限寄、时限标准、服务范围等高频政策内容，这套结构就可以稳定支撑邮政客服模型的领域数据建设。
